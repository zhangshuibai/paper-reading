TL;DR — SWE-chat 是第一个大规模、真实「野外」的 coding agent 会话数据集：通过开源工具 Entire.io 的 CLI checkpoint logging，从 200+ 个公开 GitHub 仓库收集了约 6,000 个会话、63,000+ 条 user prompt、355,000+ 次 tool call，并带有逐行的 human vs. agent 代码归属。基于它，作者给出真实使用的实证画像：coding mode 高度双峰（41% 的会话是 agent 写 >99% 代码的 "vibe coding"，23% 由人全写）；agent 越来越自主但效率不高——只有 44.3% 的 agent 产出代码最终进入 commit；vibe coding 既贵又慢，还会引入约 9× 于纯人工的安全漏洞；用户在 44% 的 turn 里通过中断/纠正/报错主动 push back。结论：agent 的自主性正在超过其寻求监督的能力，把审查负担转嫁给了用户，curated benchmark 严重低估了真实工作流的复杂度。

②研究动机与贡献 Motivation & Contribution

背景与问题

AI coding agent（在 LLM 上加一套 tool call——编辑文件、跑 shell、调 subagent）已经被大规模采用，甚至被报道能解决人类要花 12 小时的编码任务（METR 在 50% 成功率下的估计）。开发者正在以前所未有的规模把编码工作「外包」给 agent。但本文指出一个尖锐的空白：我们对人和 agent 究竟如何交互、agent 的产出到底有多少被真正采用、它们多久失败一次、用户怎么应对，几乎全是道听途说（anecdotal）。

已有评测的具体痛点在于：

  • Benchmark 只盯着 patch generation。SWE-bench、SWE-bench Multimodal、Multi-SWE-bench、SWE-bench Pro 等绝大多数 SE benchmark 都是「一组有明确可验证答案的 curated 问题」，最近的工作只是在加大难度（Terminal-bench、APEX-SWE 等），却忽略了 human-agent 交互这一维度。
  • curated 表现不等于真实可用。在打磨好的 GitHub issue 上拿高分，并不能迁移到真实的、迭代式的使用——真实场景里用户看到输出后会反复改口、纠正、打断。
  • 没有公开数据捕获完整 agentic 会话。已有数据集要么只有 agent 轨迹没有真实用户 prompt（如 SWE-smith-trajectories、CoderForge、SERA、nex-agi-agent-sft、Agent Trajectories、Multi-SWE-bench_trajs、Agent Data Protocol、AIDev），要么有代码归属但缺 tool-use 轨迹（AgentPack）。没有一个同时具备「真实用户 prompt + agent tool-use 轨迹 + code diff + 逐行代码归属」四要素（见论文 Table 1）。

作者认为，要造出真正有用的 agent，前提是搞清开发者在实践中如何使用它们，而唯一办法就是在野外收集真实使用数据。两个研究问题：

  • RQ1：真实编码任务中，用户如何与 coding agent 交互？
  • RQ2：coding agent 在实践中如何失败，用户又如何应对？

本文的切入点

关键的「使能」观察是：Entire.io 这个开源 CLI 工具，会把每一次 coding agent 会话的 checkpoint 记录到仓库的一个专用分支，并把每个 checkpoint 关联到一个带逐行 human/agent 归属的 commit。这给了作者一个前所未有的抓手——只要扫描所有 opt-in 的公开仓库，就能拿到「完整交互轨迹 + 代码归属」这种以前拿不到的数据。在此之上，作者再用 LLM-as-a-judge 对会话做语义标注（成功度、用户画像、prompt 意图、用户 push back），从而能问「怎么交互」和「为什么成功/失败」这类聚合统计回答不了的问题。

核心贡献

  1. 提出 SWE-chat 数据集——第一个来自真实用户、真实仓库的大规模 coding agent 会话数据集（约 6K 会话、63K+ prompt、355K+ tool call、2.7M 日志事件，覆盖 200+ 仓库、五种主流 agent），含完整交互轨迹与逐行 human/agent 代码归属，且为持续增长的「活数据集」。
  2. 给出真实使用的交互画像（回答 RQ1）：用户用 agent 做的事远不止写补丁——「理解已有代码」是最常见意图（19.0%），agent 有 1/3 的 tool call 在跑 bash 而非编辑文件；coding mode 极度双峰，且 vibe coding 三个月内份额从 ~20% 翻倍到 >40%；用户多为「expert nitpicker」（39.7%），交互是多轮、挑剔、迭代的。
  3. 给出失败模式与用户响应的实证（回答 RQ2）：定义了一套可从原始日志直接算出的效率/安全指标，量化发现 agent powerful but brittle——只有 44.3% 产出代码存活进 commit；vibe coding 比协作模式贵约 3×、引漏洞约 9× 于纯人工；agent 极少主动停下来提问（Claude Code 仅 1.4% 的 turn），而用户在 44% 的 turn 主动 push back。据此提出用 SWE-chat 构建真实工作流 benchmark、改进交互设计、训练 user simulator 的研究路线图。

③方法 Method

本文的「方法」是一条数据采集 + 富标注 + 指标计算的流水线，而非模型训练。整体上分三层：(a) 从 opt-in 的公开仓库自动发现并解析 Entire CLI 记录的原始会话日志；(b) 用 LLM-as-a-judge 给会话/prompt 打语义标签（成功度、用户画像、意图、push back）；(c) 直接从原始日志和代码归属数据算出一套效率与安全指标。下图是采集流程与数据规模的总览。

1. 数据采集流水线（Appendix C.1）

原始日志存放在每个仓库的 entire/checkpoints/v1 分支里，含 checkpoint/session 元信息、user prompt 和完整对话转录。流水线通过查询 GitHub Code Search API 发现所有启用了 Entire 的公开仓库，逐仓库下载 checkpoint 目录，并把原始转录解析成结构化的表（user prompt、agent response、thinking trace、tool call、tool result、逐 turn token 用量、tool-call 元信息含文件路径与 shell 命令）。数据来自五种主流 agent——Claude Code、OpenCode、Gemini CLI、Cursor、Factory AI Droid——其中实践中 ~85% 来自 Claude Code（它是 Entire CLI 最早支持、也最常用的 agent）。Entire.io 自 2026-02-10 公开发布后开源开发者迅速涌入，因此数据呈陡峭增长（图 1 左）；Entire 自家仓库（entireio/cli）的占比已随采纳增长降到 <20%。

选择偏差（很重要） SWE-chat 只覆盖主动 opt-in、使用公开仓库的 Entire CLI 用户，因此反映的是「早期采用者」群体，不一定能推广到所有 coding agent 用户，也不覆盖私有企业代码库。此外，被用户整个放弃的失败会话往往没有 commit，因而不会被记录——这会系统性地高估成功率与 agent 效率。

2. LLM-as-a-judge 富标注（Table 2 / Appendix E）

作者在四个维度上给数据打标签，全部用 LLM judge 标注全量数据：

  • Session success（0–100）：会话级，输入完整对话 + tool call 摘要，按五个维度（目标完成度、终态、agent 效率、代码/commit 质量、用户体验）给一个整数分。用 claude-sonnet-4-6 标注。
  • User persona：会话级，把用户分为 expert nitpicker（保持目标不变、不断做精确纠正）、vague requester（指令宽泛、把决策交给 agent）、mind changer（中途改目标）或 other。用 gpt-5.4-2026-03-05（reasoning_effort=low）。
  • Prompt intent：prompt 级，分为 create / refactor / debug / understand / connect / git / test / other 八类。用 Qwen/Qwen3.5-27B。
  • User pushback：prompt 级（非中断），分为 correction（纠正/补信息/改方向）、rejection（明确撤销/拒绝）、failure_report（报告产出坏了）、non_pushback。用 Qwen/Qwen3.5-9B，输入含到该 prompt 为止的完整上下文。

标注方法学很严谨：先两位标注者迭代到对 10 条全一致以得到 codebook，再各自独立标 90 条算 inter-annotator agreement，最后讨论出 100 条 gold label；然后用这 100 条 gold 在 9–11 个候选 LLM × 2–4 个 prompt 改写下做选型，挑最优模型 + prompt。作者明确警告 LLM judge 会犯错、不是可靠的标注者，选这条路只是为了「随数据增长持续标注」的可扩展性，并建议下游使用前再验证。

3. 效率与安全指标（Appendix C.2 / D.5）

这套指标全部无需标注、直接从原始日志算，是本文量化结论的基石。代码归属由 Entire CLI 在 commit 时用 shadow branch 上的临时 checkpoint 计算，得到逐行的 human/agent 归属：

$$\text{Agent-authored \%} = \frac{\text{agent lines survived}}{\text{total committed lines}}\times 100$$

其中 agent lines survived 是最终 commit 里仍归属于 agent 的行数，分母是 commit 的总行数——这个比例就是用来切分 coding mode 的依据。

为衡量「agent 白干了多少」，作者把每个改动文件还原成三态——base（父 commit）、agent actions（按时序回放每次改文件的 tool call）、committed（最终态）：用 Python 的 difflib.SequenceMatcher 在每次 tool call 后做逐行 diff，给每行打 provenance 标签（base / agent），追踪所有 agent 的增删改，再看 commit 时哪些存活。由此得两个率：

$$\text{Coding efficiency} = \frac{\text{agent lines survived}}{\text{agent cumulative lines produced}}\times 100$$

$$\text{Code survival rate} = \frac{\text{agent lines survived}}{\text{agent lines in final state}}\times 100$$

Coding efficiency（编码效率）= agent 全部产出里最终进 commit 的比例（包含 agent 自己后来又重写掉的行，所以会「惩罚」agent 的自我返工）；Code survival rate（存活率）= 在 agent 自我覆盖之后、人类保留下来的比例（不惩罚 agent 自我覆盖）。直觉上，efficiency 衡量「整体白干程度」，survival 衡量「人类对 agent 净产出的接受程度」。

此外还有一组「每 100 行 committed 代码消耗」的资源指标（lower is better）：

$$\text{Token eff.}=\tfrac{\sum \text{tokens (in+out+cache)}}{\text{committed lines}}\times100,\quad \text{Cost eff.}=\tfrac{\sum_{\text{API call}}\text{tokens}\times\text{price}_{\text{model}}}{\text{committed lines}}\times100$$

$$\text{Cognitive eff.}=\tfrac{\sum \text{user prompt chars}}{\text{committed lines}}\times100,\quad \text{Time eff.}=\tfrac{\sum \text{session runtimes}}{\text{committed lines}}\times100$$

其中 cognitive efficiency 用「用户敲的 prompt 字符数」近似用户的认知投入；time efficiency 统计完整会话时长但剔除 >2 分钟的纯空闲段。由于一个会话可能跨多个 commit、多个会话也可能贡献到同一 commit，涉及 commit 级归属的分析（Table 3、图 7）只保留能无歧义归属的会话（占 48.6%）。

安全维度则用开源静态分析器 Semgrep（--config=auto 规则集）：对每个 commit 取改动前后的仓库快照各扫一遍，只保留「commit 修改过的文件」里的发现，把「post 有、pre 没有」记为 introduced（引入漏洞）、「pre 有、post 没有」记为 fixed（修复漏洞），按每 1,000 行换算成率，并对照 CWE 分类。

④实验评估 Evaluation

本文没有「打榜」意义上的实验，而是用 SWE-chat 对真实使用做描述性实证分析，分别回答 RQ1（怎么用）与 RQ2（怎么失败、怎么应对）。下图把所有核心发现浓缩在一张图里。

评测设置

  • 数据 / 群体：SWE-chat 全量会话（~6K，~85% 为 Claude Code），分析对象是真实开发者在公开仓库的多轮交互。
  • 没有传统 baseline：这是描述性研究，「对比」体现在三种 coding mode 之间，以及与同期野外测量工作（McCain et al. 2026、METR）的趋势对照。
  • 关键指标方向：coding efficiency / code survival rate 越高越好；token/cost/cognitive/time efficiency（每 100 行消耗）越低越好；vulnerabilities introduced per 1K lines 越低越好；session success（0–100）越高越好。

RQ1 主要结果：真实使用画像

  • 任务远超「写代码」：user prompt 的意图分布里，最常见的具体意图是「理解已有代码/行为」19.0%，其次创建新代码 13.4%、git 操作 13.4%、debug 13.0%，refactor/test/connect 更少（还有 26.6% 落入宽泛的 other）。这些「代码理解 + 例行开发」能力在现有 benchmark 里被严重低估。
  • agent 一个 turn 内狂调工具：所有 tool call 里 bash 占约 1/3（以 git 操作为主），其余主要是 read/edit/grep；轨迹典型是先 read/search 再改文件、跑 build（图 21）。Table 5 给出细分：read 19.8%、edit 19.6%、git/gh 11.9%、grep 10.1%、bash:build 8.0% 等。
  • coding mode 极度双峰（核心发现，图 5）：虽然所有 committed 行里 55.8% 由 agent 所写，但分布是双峰的——
    • Human-only（22.7% 会话）：commit 代码全由人写，agent 只当理解/调试/git 助手。
    • Collaborative（36.5% 会话）：人和 agent 共同贡献（agent 写 >0% 但 <99%）。
    • Vibe coding（40.8% 会话）：agent 写了 >99% 的 commit 代码。
    三个月观察窗内，vibe coding 份额从 ~20% 翻倍到 >40%（图 25）；但「完全自主一次性解决问题」仍远非现实——交互通常跨多轮。
  • 用户多为「挑剔的专家」：persona 分布为 expert nitpicker 39.7%、vague requester 33.5%、other 19.9%、mind changer 7.0%（图 24）。即使在 vibe coding 会话里也有 47% 是 expert nitpicker；mind changing 在 vibe coding 里更少（5% vs. 其它模式 10%）。这与「benchmark 一次性给全指令」的假设截然相反——真实用户是看到输出后迭代细化。

RQ2 主要结果：失败模式与用户响应

(1) 大部分会话成功，但长尾失败有信息量。session success 分布左偏，90% 的会话拿到 50+ 分，均值 75.7、中位数 82.0（图 6）。human-only 会话的平均成功分略低于协作/vibe。作者手动检查了成功分最低（2–15）的 50 个会话，最常见的失败是：用户在 agent 产出有意义结果前就中断、以及 agent 产出/commit 的东西与用户真实诉求完全无关。

(2) agent 效率不高——近一半产出被丢弃（Table 3）。只有 44.3% 的 agent 产出代码存活进 commit。vibe coding 会话用户更「照单全收」（平均 commit 59% 的 agent 行），但这难以因果归因——可能是产出更对路，也可能是用户审查更松。

标注质量验证（消融性证据，Appendix E）

由于结论高度依赖 LLM judge，作者专门验证了标注可靠性。Inter-annotator agreement（Table 6/7）：prompt intent Cohen's κ=0.709（一致率 75.6%）、prompt pushback κ=0.832（二分类版 κ=0.888）、user persona κ=0.662（78.9%）、session success Spearman ρ=0.757 / ICC(2,1)=0.503——「moderate-high」。LLM 对 100 条 gold 的表现（Table 8）：所选模型在 prompt intent 0.76（qwen-3.5-27b）、user persona 0.69（gpt-5.4）、repository domain 0.81 / audience 0.84（claude-opus-4-6）、pushback binary 0.79（qwen-3.5-9b）、session success ICC(2,1)=0.60（claude-sonnet-4-6）。作者据此主动声明这些标签不应被当作 ground truth，仅用于「在海量数据里方便地筛例子」。

⑤洞见 Insights

Insight 1 · 自主性正在超过监督能力 最反直觉也最值得记住的一点：agent 的自主性增长速度，快于它「学会何时寻求指导」的速度。一边是 >40% 的会话已是 vibe coding、turn 时长长尾突破 100 分钟；另一边 agent 只在 1.4% 的 turn 主动提问，而用户却要在 44% 的 turn 里手动顶回去。换句话说，「catch 不安全/跑偏」的负担几乎完全转嫁给了人类——而人类的审查恰恰是错误唯一被发现的途径。
Insight 2 · powerful but brittle，且「协作」才是甜点区 agent 写了一半以上的 commit 代码，但只有不到一半的产出存活——更多自主没有换来更高效的交付。而 human-agent 协作（collaborative）模式在几乎所有效率指标上都是最划算的（成本 \$0.05/100 行最低、时间最快），这暗示「全自主」的方向可能适得其反。这不是反对用 coding agent，而是说它们远没有发挥出应有的效率。
Insight 3 · curated benchmark 系统性低估了真实复杂度 真实使用里最常见的意图是「理解代码」而非「写代码」，交互是多轮、挑剔、不断改口的；而主流 benchmark 是「一次性给全指令、只评 patch」。这种错配意味着排行榜上的强表现不能简单等同于真实可用——下一个前沿在「协作工作流」的评测，而非更难的单点任务。

局限与未来

作者自陈的局限相当坦诚：

  • 选择偏差：只覆盖 opt-in、用公开仓库的 Entire CLI 用户（早期采用者），不含私有企业代码；此刻结论未必能推广。随采纳增长会更多样化。
  • 失败被低估、贡献也被低估：被整段放弃的会话因无 commit 而未被记录 → 高估成功率与效率；同时把「人换个文件重写 agid 建议」算作 human 删除 → 又可能低估 agent 的真实有用性（逐行归属抓不住语义层面的复用）。
  • 指标是代理：committed lines、prompt 字符数等都只是用户满意度的代理，不应直接当优化目标。
  • LLM judge 不可靠：明确建议下游再验证，不要照单全收标签（并引用作者自己关于「LLM annotation hacking」的工作）。

未来方向（Section 5.1 路线图）：① 用真实会话轨迹构建扎根真实工作流的 benchmark（如评 agent 在真实对话上下文里是否提出合适的下一步）；② 设计更自适应的 human-agent 交互（让 agent 更会在合适时机求澄清）；③ 用真实交互轨迹训练 user simulator 以支持离线评测——这是把 SWE-chat 当作「原料」的关键应用。作为「活数据集」，它还能支持对 agent 演化的纵向（longitudinal）分析。

与相关工作的关系（基于本文自身论述）

本文把自己定位在「野外 agent 评测 / human-agent 交互研究」这条新兴线上，与三类工作区分：① SE benchmark（SWE-bench 系列、Terminal-bench、APEX-SWE 等）——它们 curated、只评 patch，本文正是要补上交互维度；② 已有 agent 数据集（SWE-smith-trajectories、AgentPack、Agent Data Protocol 等）——本文 Table 1 显示唯有 SWE-chat 同时具备真实用户 prompt + tool-use 轨迹 + code diff + 代码归属；③ 野外测量类工作（McCain et al. 2026 的 agent autonomy、METR 的 time horizon、Anthropic 的经济指数报告）——本文在多处与其趋势对照（如 turn 时长、停止/push back 率），并补充了它们没有的「逐行代码归属 + 完整交互轨迹」。安全方面则把「LLM 会写不安全代码」（Pearce et al.、Perry et al.、Fu et al.）从 synthetic 设定延伸到真实 coding agent 工作流，并指出现有安全缓解手段（security-hardening fine-tune、system-prompt hardening）大多只在 synthetic benchmark 上验证过，SWE-chat 恰好提供了真实测试床。
