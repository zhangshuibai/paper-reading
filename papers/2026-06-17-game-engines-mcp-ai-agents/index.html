一句话 — 「AI 做游戏」实际上是三件不同的事：(a) AI agent 驱动现有引擎（让 Claude/Cursor 经 MCP 去 Unity/Unreal/Godot/Blender 里建场景、写脚本、调参）；(b) 零代码 text-to-game（Rosebud / Astrocade / HYTOPIA 等，一句话生成可玩小游戏）；(c) 端到端 generative game / world model（DeepMind Genie 3、Decart Oasis、Microsoft Muse，神经网络直接生成可玩画面，根本不经过传统引擎）。现状：路线 (a) 的 MCP server 几乎全是社区项目（unity-mcp 已被 Coplay 收购、unreal-mcp、blender-mcp 最成熟），官方仅 Unity 把 AI 收进编辑器（Unity AI，已替代被弃用的 Muse）；路线 (b) 能做 demo / 小游戏但难做「正经游戏」；路线 (c) 仍偏 research / 早期产品。真正成熟的「游戏生成的 Cursor」还不存在——最大 whitespace 在「把 MCP 接进 UE/Unity，做成会自己搭场景、调参、写玩法、还能自测的成熟 agent」，以及垂直品类的 text-to-game。

①领域背景速补（假设你只懂 LLM、不懂游戏引擎）

为了让只有 LLM 背景的读者跟上，先把几个最容易混的概念拆清楚。

游戏引擎 vs 纯渲染引擎

渲染引擎（renderer）只负责「把 3D 场景算成一帧画面」——光照、阴影、材质、光栅化 / 光线追踪。它是「画图的」。游戏引擎（game engine）是个大全套，渲染只是其中一块，还包含：物理引擎（碰撞、重力、布料）、脚本 / 逻辑系统（C#、GDScript、蓝图，决定「按下跳跃键角色怎么动」）、资产管线（导入模型、贴图、动画、音频）、编辑器（拖拽搭关卡的 GUI）、以及打包发布到各平台的工具链。类比 LLM 世界：渲染引擎像「单独一个 attention kernel」，游戏引擎像「整个 PyTorch + 训练框架 + 数据管线 + 部署」。本文主角是游戏引擎；Blender 是 3D 建模 / 资产工具（不是游戏引擎），但它是 AI 做游戏「资产侧」最重要的一环，所以一并盘点。

MCP（Model Context Protocol）：让 agent 调工具的「USB 口」

MCP 是 Anthropic 2024 年底提出的开放协议，标准化了「LLM / agent 如何调用外部工具和数据」。把它想成 AI 的 USB-C：一边是 MCP client（Claude Desktop、Claude Code、Cursor、VS Code 等会「说 MCP」的 AI），另一边是 MCP server（一段程序，把某个软件的能力包装成一组「工具」暴露出来）。「Unity MCP server」= 一段把 Unity 编辑器的操作（建物体、改材质、写脚本、跑测试）包成 MCP 工具的程序，这样任意 MCP client 都能用自然语言驱动 Unity。本文路线 (a) 的核心就是「各引擎有没有 MCP server、是官方还是社区、能干啥」。

CLI / headless：没有界面也能跑

CLI（命令行接口）指能用命令行参数驱动引擎而非点 GUI；headless（无头）指不开窗口、不初始化图形设备地运行（如 -nographics、--headless），常用于服务器、CI 自动构建、批量导出。对 AI agent 很关键：agent 不会用鼠标点 GUI，它需要能通过命令行 / API 程序化地驱动引擎。一个引擎的 CLI / 脚本 API 越完整，越容易被 agent 接管——MCP server 往往就是封装在这些底层能力之上的。

「AI 做游戏」的两条大路线，务必分清

  • (a) AI agent 驱动现有引擎：AI 生成代码 / 资产 / 关卡配置，最终仍由传统引擎（Unity/Unreal/Godot）渲染和运行。游戏的「确定性、可调试、可发布」由引擎保证，AI 只是「会写代码 / 会搭场景的助手」。这条路线又分两支：引擎内的 Copilot/Cursor 式助手（Ludus AI、Unity AI），和把外部 AI 经 MCP 接进引擎。
  • (b) generative game / world model：一个神经网络直接输出可玩的画面帧，玩家的操作（键鼠）作为输入喂进去，模型预测下一帧——没有传统意义上的代码、关卡文件、物理引擎，整个「游戏」就是模型权重。代表：DeepMind Genie、Decart Oasis、Microsoft Muse/WHAM、World Labs。优点是「想象即可玩」；缺点是当前只能维持几十秒到几分钟一致性、分辨率有限、难精确控制、几乎无法做成「可发布、可联机、规则确定」的正经游戏。
一句话区分路线 (a) 是「AI 当程序员 / 美术，引擎照常跑游戏」；路线 (b) 是「模型就是游戏，引擎被整个跳过」。媒体常把两者混为「AI 生成游戏」，但它们的成熟度、可控性、商业形态完全不同。

③MCP / CLI 接口现状（逐引擎核实带链接）

这是路线 (a) 的「管道层」。一个关键结论先放这：截至目前，几乎所有引擎 MCP server 都是社区项目，没有一家大引擎厂商官方发布过自家 MCP server。官方动作集中在「把 AI 收进自家编辑器」（Unity AI），而非「开放 MCP 让外部 agent 接管」。下面逐个核实。

Blender — 🟢 社区旗舰，最成熟

ahujasid/blender-mcp（Siddharth Ahuja，开源）—— 当前最知名、最常被引用的引擎类 MCP server。架构是：一个 Blender 插件（addon.py）在 Blender 内开 socket 服务，加一个 MCP server（server.py）说 MCP、连到插件。能力：创建/修改/删除 3D 物体、改材质和颜色、查询场景信息、在 Blender 内执行任意 Python 代码。MCP client 支持 Claude Desktop、Cursor、VS Code 等。🟢 一手仓库

Unity — 🟡 社区成熟（已被收购），官方走 Unity AI

CoplayDev/unity-mcp（MIT 开源）—— 原由 justinpbarnett 开发，后被 Coplay 收购并继续维护。在 Unity 编辑器和 AI 助手间架桥，暴露 40+ 工具：manage_scene、manage_script、manage_gameobject、manage_material、run_tests 等，外加 25+ 只读资源做状态自省；自动配置 Claude Desktop / Claude Code / Cursor / VS Code / Windsurf / Cline / Codex / Gemini CLI 等 client。🟢 一手仓库

官方侧：Unity AI —— 注意一个重要变化：原 Unity Muse（$30/月的 AI 助手）已被弃用，于 2025-10-01 退役，由 Unity AI（随 Unity 6.2 推出）替代。Unity AI 改用第三方前沿模型（不再只用 Unity 自研），内嵌编辑器、带完整工程上下文，按 Unity Points 计费而非单独订阅。🟡 官方博客/媒体 这是「引擎内 Copilot」路线，不是 MCP。

Unreal Engine — 🟡 多个社区实验，标注 EXPERIMENTAL

chongdashu/unreal-mcp（社区，约 1.9k★）—— 让 Cursor / Windsurf / Claude Desktop 用自然语言控制 UE。工具含：创建/删除 actor、设置 transform、查询 actor 属性、创建带组件的 Blueprint 类、设置组件属性与物理。配套 UE 5.5 空白起始工程已带 UnrealMCP.uplugin。仓库自我标注为 EXPERIMENTAL，API 仍会大改。🟢 一手仓库

另有 ChiR24/Unreal_mcp（TypeScript + C++ 原生 Automation Bridge 插件）、kvick-games/UnrealMCP 等多个并行实现。🟡 二手聚合页 底层依赖：UE 的 Remote Control API（HTTP/WebSocket 远程控制）+ 编辑器 Python，这是 agent 能接进 UE 的根基。

Godot — 🟡 社区项目最多（碎片化）

Godot 因开源 + headless CLI 友好，社区 MCP 实现数量最多但很碎片化，质量参差。已核实存在的包括： Coding-Solo/godot-mcp（启动编辑器、运行工程、抓调试输出）、 tugcantopaloglu/godot-mcp（自述 149 工具，全引擎控制）、 IvanMurzak/Godot-MCP（C#，Apache-2.0，连云端 ai-game.dev）、 mkdevkit/godot-mcp、 bradypp/godot-mcp 等。🟢 一手仓库 没有 Godot 官方 MCP。底层 CLI：godot --headless --export-release … 支持 CI 导出。

O3DE — 🟡 单个社区实现

nickschuetz/o3de-mcp —— 自动化 Open 3D Engine 的 MCP server，做编辑器控制、工程与构建管理。O3DE 本身自带 CLI 工具，开源（Apache 2.0）对 agent 友好。🟢 一手仓库

未找到知名 MCP server 的引擎 🔴

CryEngine、Cocos、Construct、GameMaker、Defold、Bevy 截至核实未找到广为人知的官方或社区 MCP server（不排除存在零散个人项目）。其中 Cocos / Bevy / Web 引擎（Three.js 等）因 API 全代码化，更多是被通用代码 agent（Cursor/Claude 直接写 JS/Rust）驱动，而非专用 MCP。🟡 检索为负，标「未找到」而非「不存在」

这一层的核心判断MCP/CLI 管道已经能用，但全靠社区、且大多 EXPERIMENTAL：blender-mcp 最稳，unity-mcp 因被收购有商业维护，unreal-mcp/godot-mcp 仍在快速变动。瓶颈不在「能不能连上引擎」，而在「连上之后 agent 能不能可靠、可重复、可自测地完成一个完整玩法」——这一步目前都还脆弱。

④「AI 做游戏的 agent / 游戏生成的 Cursor」landscape

把玩家按前述三条路线归类。每个标 research / 产品、官方 / 社区，并按来源分级。

路线 (a-1)：引擎内 AI 助手 / Cursor 式（agent 驱动现有引擎）

  • Ludus AI（Unreal）—— 把 AI 嵌进 UE5 的插件：LudusChat（自然语言生成 3D 场景/资产）、LudusBlueprint（蓝图 copilot）、LudusCode（C++ 助手）、LudusDocs。支持 UE 5.4–5.7；蓝图 Agent 生成 2025-07 进 Open Beta。🟡 官网/媒体 这是目前最接近「Unreal 里的 Cursor」的产品之一。
  • Unity AI（官方，Unity 6.2+）—— 内嵌编辑器的 Assistant + Generators，用第三方前沿模型，带工程上下文。替代已退役的 Muse。🟡 官方
  • 把 Cursor / Claude 经 MCP 接进引擎 —— 即第 ③ 节那些 MCP server。本质上是「用通用 coding agent + 引擎 MCP」拼出一个引擎内助手，灵活但需自己搭、稳定性看社区项目。🟢 一手仓库

路线 (a-2)：text-to-game 零代码平台（AI 生成代码/资产，引擎仍跑游戏）

  • Rosebud AI —— 无代码「vibe coding」做 3D 游戏 / 交互世界，描述场景规则即生成可玩逻辑，带模板、资产、实时预览、即时部署、内置变现与创作者社区。🟡 官网
  • Astrocade —— 文字 prompt 生成社交化小游戏（含美术、动画、特效、音乐、玩法），偏「快做、好玩但浅」，不做沉浸 3D / 长线世界。🟡 二手
  • HYTOPIA —— 浏览器端 voxel 引擎 + TS/JS SDK，AI 按 prompt 生成代码、地图、怪物、UI、计分，可直接发布可玩；还出了 MCP & AI API 插件直连 Cursor。瞄准 UGC 游戏赛道。🟡 官方博客
  • Series AI（Rho Engine） —— 「AI-native 多模态全栈游戏创作平台」，一套 copilot + AI 技术，可与 Unity/Unreal 协作（引擎无关的内容生成器，像「meta engine」）。据报道已悄悄融资约 $28M（Netflix、Dell、a16z 等，2024）。🟡 媒体 TechCrunch/GamesBeat
  • Buildbox 4 —— 老牌无代码引擎，4.0 主打「用 AI 做游戏」。🟡 官网
  • Layer —— 严格说是游戏美术资产侧（图像/视频/3D/音频，149+ 模型，可训练自有画风），不是 text-to-game，但属同一工具链。🟡 官网

路线 (b)：generative game / world model（模型即游戏，跳过传统引擎）

  • DeepMind Genie 3 —— 文本/图像/草图生成可实时交互的虚拟世界，24fps、720p、一致性可维持数分钟（Genie 2 仅约 10 秒）。2025-08 公布；2026-01-29 以 Project Genie 形式向 Google AI Ultra 订阅者（美国）开放。🟡 官方/媒体 介于 research 与早期产品之间。
  • Decart Oasis —— 把 world model 直接做成游戏（基于 Minecraft 画面训练），实时开放环境 + 涌现物理 + 流式扩散，自述「数百万用户、3 天破百万」。🔴 用户数为自述 偏 demo / 体验产品。
  • Microsoft Muse（WHAM） —— 微软研究院 + Ninja Theory 的 World and Human Action Model，能生成游戏画面 / 手柄动作，一致性约 2 分钟；用《Bleeding Edge》训练（15M–1.6B 参数），最大模型 2025-02 发表于 Nature，开源权重 + 样本数据，WHAMM 扩展迁移到 Quake II。定位「gameplay ideation 研究」，不是发售游戏。🟡 官方研究/Nature
  • World Labs —— 「大世界模型（LWM）」，从单张图生成可探索 3D 环境（几何 + 深度，可导航），偏「prompt-to-world」。已与 Rosebud 有合作案例。🟡 官网 research / 早期产品。

定位图：横轴 = 改造现有引擎 ↔ 端到端神经生成；纵轴 = 面向开发者 ↔ 面向零基础玩家。左半（蓝/绿）靠传统引擎、可控可发布但需搭管道；右半（橙/红）端到端、想象即玩但偏 demo。中间虚线框 = 最大空白：开发者与玩家之间、「会干一整套引擎活又可靠」的 agent 尚缺。（仅技术现状梳理，非投资建议 / 非背书）

⑤现状判断 + whitespace

现状三句话

  • MCP/agent 驱动引擎：管道通了，但多为社区实验、尚不成熟。blender-mcp 最稳、unity-mcp 有商业维护，unreal-mcp/godot-mcp 还在快速变动、自标 EXPERIMENTAL；官方引擎厂商至今没人发自家 MCP，只把 AI 收进编辑器（Unity AI）。
  • 「游戏生成的 Cursor」真正能用的还少。Ludus（UE）、Unity AI、Series Rho 等已是「引擎内助手」，但离「给一句话、agent 自己把一个完整可玩、可调试、可发布的游戏从场景到玩法到测试全做完」还有明显距离——能做 demo 和片段，做不了「正经游戏」的端到端闭环。
  • 端到端 generative game 仍偏 demo / research。Genie 3 / Oasis / Muse 惊艳，但受一致性时长、分辨率、可控性、无法联机/发布/规则确定所限，短期内是「体验 / 创意工具」，不是「做出可上架游戏」的路径。

最有机会的空白（whitespace）

whitespace A · 成熟的「引擎 agent」把 MCP 接进 UE / Unity，做成一个真正可靠的 agent：会自己搭场景、调参数、写玩法脚本、还能自测（跑 PIE / 单元测试、看报错、自修）。难点全在「确定性与自验证」——这正是当前社区 MCP 最弱的一环，也是把「能连引擎」变成「能交付游戏」的关键缺口。谁先把这条闭环做稳，谁就接近「真正的游戏 Cursor」。
whitespace B · 垂直品类 text-to-game通用 text-to-game（Rosebud/Astrocade）什么都能做一点、但都浅。垂直切口——某一类玩法（卡牌 / 塔防 / 视觉小说 / 教育 / 小游戏买量素材）的「专精生成器」，靠领域模板 + 资产 + 玩法约束把质量做到「能直接上线」，比通用平台更易做出可用产品。中国侧叠加 Cocos / 微信小游戏生态是天然切口。

诚实给风险

  • 引擎复杂度：UE/Unity 的状态空间巨大（蓝图、序列化、依赖、平台差异），agent 容易「看似改对了、运行崩了」。
  • 确定性 / 可重复：游戏要求规则稳定、可调试；LLM 的随机性与「自信地出错」与之天然冲突，自验证回路是硬骨头。
  • 版权 / 训练数据：生成资产与代码的版权归属、训练语料合规（Unity 已明确把版权责任推给用户）是悬而未决的风险。
  • 质量天花板：「能跑的 demo」到「好玩、耐玩、能变现的游戏」之间，是玩法设计与打磨，这部分 AI 短期补不上。
