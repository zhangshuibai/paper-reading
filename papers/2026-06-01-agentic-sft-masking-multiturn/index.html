TL;DR — ① 多轮工具调用 SFT 里，tool response（工具返回 / observation）会被 mask 掉、不计入 loss；只有 assistant 自己生成的 token（reasoning + tool call + 最终答案）算 loss——否则模型会学着自己编造工具返回。② 一条 K 轮会话怎么变成训练数据，看有没有 thinking：普通多轮 / 多轮工具调用 → 整段打成「一条」（靠 mask 同时学所有 assistant 轮）；带 thinking 的多轮 → 按轮拆成 N 条（每条删掉历史轮的 `<think>`），因为历史 thinking 在推理时不可见、单次 forward 会造成 train/inference mismatch。One-Pass to Reason 用 token 复制 + 块稀疏 mask 让「不拆」也能等价、还更省。

①一条多轮会话里，哪些 token 算 loss

把一条带工具调用的会话摊平成 token 序列后，SFT 的 cross-entropy 只算在 assistant 自己生成的部分上；system / user / tool（工具返回）的 token 全部把 label 置成 -100，被 CrossEntropyLoss 忽略。

为什么 tool response 必须 mask？ 它是环境产生的、不是模型该生成的。若把它算进 loss，等于训练模型去凭空编造工具返回值（hallucinate observations）——推理时模型就会跳过真实调用、自己瞎编 API 结果。所以只让模型学「何时调、怎么调」，把「返回什么」交给真实环境。

出处。 框架层最直白的是 LLaMA-Factory 的 ShareGPT 格式：human/observation 在奇数位、gpt/function 在偶数位，文档原话「The gpt and function will be learned by the model.」——即 assistant 的工具调用（function）要学、工具返回（observation）不学。机制层是 HuggingFace apply_chat_template(return_assistant_tokens_mask=True) 配合模板里的 {% generation %} 关键字（assistant token 标 1、其余标 0），驱动 TRL SFTTrainer 的 assistant_only_loss。

出处：LLaMA-Factory data/README · HF chat templating（tool calls vs tool responses） · TRL assistant_only_loss

连回 ACC（2605.21850） ACC 整篇的动机正是冲这个惯例来的：标准 agent SFT「把 tool response 全 mask、只监督 turn-level 的 tool selection」造成监督盲区——散落各 turn 的长程证据被浪费。ACC 把 trajectory 编译成 long-context QA 对，让答案梯度绕过 mask、直接触达每个证据 token。

②一条 K 轮会话 → 几条训练数据？两种做法

做法 A：整段打成「一条」样本（默认、高效）。 整个 K 轮拼成一个 token 序列、一次 forward 过完，loss 靠 masking 同时算在所有 assistant 轮上。共享前缀只编码一次，开销 $O(N)$。LLaMA-Factory 默认、TRL assistant_only_loss、ms-swift 都是这套。

做法 B：按 turn 拆成 N 条。 K 轮 → K 条，第 $i$ 条 =「前 $i\!-\!1$ 轮历史当 context ＋ 第 $i$ 轮 assistant 当 target」，每条只算当前轮 assistant 的 loss。

例：会话 [sys, u1, a1(调用), tool1, a2(最终)]
· 做法 A → 1 行：labels 里只有 a1、a2 激活，其余 mask。
· 做法 B → 2 行：[sys,u1]→a1 和 [sys,u1,a1,tool1]→a2。

分水岭是：有没有 thinking、且它要不要从后续 context 删掉。普通多轮 / 多步工具调用（历史原样保留）走 A 即可；带 <think> 的多轮通常被迫走 B——原因见下一节。

③为什么「带 thinking 的多轮」单次 forward 不行（细讲）

先记两个事实：

  • 事实 1：单次 forward = 整段拼成一个序列跑一遍；causal attention 下每个 token 能注意到它前面的所有 token。
  • 事实 2：reasoning 模型（Qwen3 / DeepSeek-R1）推理时，历史轮的 <think> 已被删掉，回答第 2 轮时看到的历史里只有第 1 轮的答案、没有第 1 轮的思考。即第 2 轮的真实 context = [Q1, A1, Q2]，不含 T1。

矛盾就出在这。若把整段 [Q1, T1, A1, Q2, T2, A2] 拼成一条训练（T=thinking，A=答案）：

那为什么不能只加个 attention mask、在单条里把 T1 挡住？ 卡点在 A1：第 1 轮答案 A1 本身必须看见 T1（模型「先想 T1 再答 A1」，这因果要训练）；可 A1 之后的 Q2/T2/A2 又必须看不见 T1。一条序列里只有一份 A1，它被要求同时满足两件互斥的事：① 当训练目标时带 T1 的信息；② 当第 2 轮的历史时是「不带 T1 的干净版」。一份 token 做不到 → 最简单就只能拆开，每条单独服务一个轮次。

出处：QwenLM/Qwen3 Discussion #1398——官方维护者原话「the thinking content of the previous turns should be removed except for multi-step tool calls」，并给出拆分示例 [Q1,T1,A1,Q2,T2,A2,Q3,T3,A3] → [Q1,T1,A1]、[Q1,A1,Q2,T2,A2]、[Q1,A1,Q2,A2,Q3,T3,A3]。

注意那句「except for multi-step tool calls」 多步工具调用里，中间轮的 reasoning 往往要保留（后面步骤依赖前面想法），所以纯 tool-use 的多轮常常可以走做法 A 不拆；真正逼你拆的是「历史 thinking 要删」这一条。

④One-Pass to Reason：不拆也能等价、还更省

拆成 N 条虽然正确，但把共享前缀反复重编码，时间 $O(N^3)$、很浪费。《One-Pass to Reason》（IJCNLP-AACL Findings 2025，DevRev）给了个巧解：把 A1 这种「既要被训练、又要当干净 context」的 token 复制成两份（一份带 T1 用于算它自己的 loss，一份不带 T1 用于给后续轮当历史），再配一个 block-sparse attention mask 精确控制可见性，于是一条序列、单次 forward 就能得到与 N-pass 完全相同的 loss，时间从 $O(N^3)$ 降到 $O(N^2)$、显存不变。

原文：「Fine-tuning LLMs on multi-turn reasoning datasets requires N (number of turns) separate forward passes per conversation due to reasoning token visibility constraints.」——这句正是「带 thinking 必须拆」的权威表述。

一句话：tool response 一律 mask（别让模型编造环境返回）；不带 thinking 就整段一条、带 thinking 就按轮拆 N 条（删历史思考）——这是 Qwen3 官方建议，而 One-Pass to Reason 给了不用拆也等价的高效解法。
