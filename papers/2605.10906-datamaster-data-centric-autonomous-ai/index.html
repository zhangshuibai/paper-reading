TL;DR — 当 model、训练配方、算力都越来越标准化之后，ML 的性能差异越来越多地由数据决定。本文提出一个新问题 task-conditioned autonomous data engineering：固定 learning algorithm（模型代码、超参、训练脚本都不动），让一个 agent 只在数据侧做优化——发现外部数据、选择与组合、清洗与转换、构造特征——去最大化下游表现。为此提出 DataMaster，把数据工程组织成一棵 DataTree 的树搜索（red node 探索新外部数据、black node 清洗整合并真正跑下游训练拿反馈），配一个跨分支共享的 Data Pool 和累积式 Global Memory。在 MLE-Bench Lite 上 medal rate 相对最强初始解 +32.27%（35.91%→68.18%），在 PostTrainBench 上平均分 +22.70 个点（8.47%→31.17%），并在 GPQA 上超过人类专家训练的 instruct model（31.02% vs 30.35%）。

②研究动机与贡献 Motivation & Contribution

背景与问题

ML 的进步历史上由「更好的模型、更好的算法、更多算力」驱动。但作者观察到一个趋势：模型开发本身正在被标准化——pretrained backbone 随处可用、训练配方易于复用、开源栈让 fine-tune 越来越「即插即用」、评测流水线也更可复现。于是性能差异越来越多地由数据塑造：用什么数据、怎么选怎么清洗、怎么混合、与下游目标匹配得多好。这把 ML 推向一个更强的 data-centric 阶段，也改变了「自主系统应该优化什么」这件事。

然而当前的自主 ML 系统（autonomous ML / ML engineering agents）仍然是 model-centric 的：像 AIDE、SELA、MLZero、MLE-STAR、ML-Master 这些 agent 擅长写代码、调 debug、调超参、从 validation 反馈迭代，但它们把训练数据当成固定给定（fixed given），action space 局限在模型代码/架构/超参，数据本身从不被优化。更新的 PostTrainBench 虽然允许 agent 在 post-training 时 curate 数据，但数据 curation 仍被当作一个辅助性的 action 维度，而非优化的中心对象。data-centric 一侧已有的工作（如 CAAFE 做 feature engineering、Co-STEER、Dataforge、LAMBDA、DatawiseAgent，以及 Spider2-V、KramaBench 等 benchmark）则各自只覆盖流水线的一部分，而且很少被下游训练反馈串起来——它们大多假设数据源「事先给定」，agent 只负责处理给定的数据集。

作者把「在数据侧系统性提升下游表现」这件事单独拎出来，定义为 task-conditioned autonomous data engineering，并指出它比「多搜几个数据集拼到训练集里」要难得多，难点有三：

  1. 数据搜索是开放式的（open-ended）。 有用的数据散落在各种 repository 和开放网络上，质量参差、license 不清、与任务的相关性高度可变；搜索空间无法事先枚举。
  2. 发现的数据很少能「即插即用」。 schema 对齐、label 映射、格式转换、过滤、去重，往往要先合成处理，数据才能进训练。
  3. 数据迭代本质是累积的（cumulative）。 为避免重复走失败方向、并在过去成功上继续建构，agent 必须记住「在哪搜过、试过哪些 transformation、哪些 adapter 有效、什么数据真的提升了表现」。

本文的切入点

作者的关键判断：上面三个难点（open-ended 搜索、branch-dependent 的精修、delayed validation）天然对应「树 + 记忆」的结构。数据迭代是可分支的（branchable）、路径依赖的（path-dependent）、累积的，而且只有通过真实下游训练才可验证。因此把数据工程显式建成一棵在「可执行数据状态」上做的树搜索，并辅以跨分支共享的候选数据层和持久记忆，就能把原本零散的数据操作变成一个有结构的搜索过程——让 agent 复用成功选择、避开重复失败、把有限预算花在更有希望的方向上。

核心贡献

  1. 提出问题： 将 task-conditioned autonomous data engineering 形式化为一等研究问题——在固定 learning algorithm 的前提下，把数据状态（而非模型代码）作为主要优化对象，用下游训练表现作为唯一质量信号来驱动搜索。
  2. 提出方法： 设计 DataMaster，一个 memory-augmented 的 data-agent 框架。其 DataTree 把自主数据工程组织成对「数据状态」的树搜索（red node 探索外部数据 / black node 数据精修并下游验证），配以跨分支共享的 Data Pool 和累积复用的 Global Memory；在有限评测预算下用 growth policy + UCB1 风格的 scheduling policy 控制树的生长与调度。
  3. 系统验证： 在两个互补 benchmark 上评测——MLE-Bench Lite（经典 ML 工程，提供初始数据）与 PostTrainBench（LLM post-training，不提供任何训练数据，全靠 agent 从零发现）。结果显示「只动数据侧」即可在两种设定下都显著提升强初始解：MLE-Bench Lite medal rate +32.27%（→68.18%）、PostTrainBench 平均分 +22.70 点（→31.17%），并在 GPQA 上超过人类专家训练的 instruct model。

③方法 Method

问题形式化

定义一个数据状态 $D$ 为在固定 learning algorithm 下、由数据侧操作产生的可执行配置：它指定了可用数据源、被选中并组合的训练子集、以及把它们转换成算法最终消费的输入所需的处理逻辑。问题输入是：下游任务 $T$、固定算法 $a_0$、初始可执行数据状态 $D_0$、以及一个资源预算 $B$（按时间或算力度量）。对任意候选数据状态 $D$，用 $E(T,a_0,D)$ 表示其下游表现。agent 在从 $D_0$ 出发、预算 $B$ 内可达的数据状态集合 $\Pi_B(D_0)$ 上搜索，终止于表现最优的精修状态 $D^*$：

$$ D^{*} = \arg\max_{D\in\Pi_B(D_0)} E(T, a_0, D). $$

这里 $a_0$ 始终不变，$D^*$ 是最终喂给算法产出下游模型的数据配置。这个问题难在两点：open-world——有用数据可能来自任意外部 repository 和网络，搜索空间无法事先枚举；long-horizon——一个数据侧决策的价值往往要等到后续精修和真正训练之后才显现。因此求解既需要广泛的候选探索，又需要对过去发现/转换/结果的累积记忆。

整体框架

DataMaster 用一个自主 agent 来近似上述搜索：不做一次性的数据集 curation，而是增量地搜新数据、试不同精修选择、并用下游训练反馈更新后续决策。这要求三种能力——从同一当前状态尝试多个下一步、跨分支共享发现的外部数据集、复用早先尝试学到的东西——分别由三个耦合组件承载（见图 1）：DataTree（树搜索骨架）、Data Pool（共享候选数据层）、Global Memory（持久记录层），再加上树的生长与调度机制让有限预算的搜索更高效。

DataTree：在可执行数据状态上的树搜索

形式上 DataTree 记为 $\mathcal{T}=(\mathcal{V},\mathcal{E})$：根节点 $v_0$ 关联初始数据状态 $D_0$；$\mathcal{V}$ 是已执行的数据工程节点集合；每条边 $\mathcal{E}$ 表示父到子的上下文继承，让子节点能引用父节点累积的数据状态、产物、决策与下游反馈，再做下一步数据侧决策。每个节点代表 agent 在当前分支上下文中执行的一步数据工程。DataTree 含两类节点：

  • Red Node（数据探索 / Data Exploration）。 负责拓宽搜索——为下游任务发现可能有用的外部数据集。给定当前搜索上下文、共享 Data Pool $\mathcal{P}$ 和从 Global Memory 取回的记忆 $\mathcal{M}_v$，red node 先根据任务描述、目标 metric、数据 schema、失败案例与历史记忆生成搜索 query，做开放式外部搜索 + 轻量筛选（看 metadata、schema 兼容性、modality、规模、label 结构、样本预览，并过滤掉重复源和 benchmark 受限源），输出一批新发现的数据条目 $\Delta\mathcal{P}_v$ 追加进 $\mathcal{P}$，供之后 black node 利用。每个保留条目存一份紧凑 manifest：本地路径/源指针、格式、schema 摘要、任务相关性、筛选备注。这一步把「开放世界外部搜索」变成一个可复用的候选层，把数据获取与下游精修解耦。
  • Black Node（数据利用 / Data Exploitation）。 负责加深搜索——利用 Data Pool 中的候选来改进当前训练数据流水线。它从 $\mathcal{P}$ 中选有用的源、调整组合、应用清洗与转换规则、构造特征、并适配到固定下游训练接口。这些决策被物化为一个可执行的 DataLoader 及相关数据处理逻辑，产出该节点特定的数据状态 $D_v$，再通过固定的 submit 接口提交、在固定算法 $a_0$ 下实例化下游评测 $E(T,a_0,D_v)$，得到 score $y_v$ 与诊断 $\phi_v$ 写回 $\mathcal{M}$。换言之，black node 把「候选数据集」转成「可执行训练假设」，并用真实下游训练来比较、诊断、精修数据侧决策，而非仅靠启发式选择。

Global Memory & Data Pool：共享搜索状态

Global Memory $\mathcal{M}$ 是 DataTree 的持久记录层，对每个已执行节点维护一条一一对应的记录，追踪分支级搜索历史、节点输出、下游结果与可复用发现：

$$ r_v = \begin{cases} (\texttt{red},\ \Delta\mathcal{P}_v,\ \phi_v), & v\text{ 是 red node},\\[4pt] (\texttt{black},\ D_v,\ y_v,\ \phi_v), & v\text{ 是 black node}. \end{cases} $$

对节点 $v$，默认取回的上下文继承自其父节点与兄弟节点：$\mathcal{M}_v = \{r_{\mathrm{Par}(v)}\}\cup\{r_u \mid u\in \mathrm{Sib}(v)\}$，其中 $\mathrm{Par}(v)$ 是父节点、$\mathrm{Sib}(v)$ 是兄弟集合。除默认取回外，agent 还可主动 query 其他 red/black 节点记录，并把新发现写回 $\mathcal{M}$。这让搜索树「相互协调」而非各自独立：red node 避免重复发现并瞄准未覆盖方向，black node 复用已验证状态、迁移成功的处理选择，更高效地花训练预算。

Data Pool $\mathcal{P}$ 是共享候选数据层，把 red node 发现的外部源转成 black node 可复用的候选。每个条目存下游使用所需信息：本地路径、源描述、格式、schema、metadata、规模、modality、任务相关性、轻量筛选结果。red node 把 $\Delta\mathcal{P}_v$ 追加进 $\mathcal{P}$，black node 从 $\mathcal{P}$ 读取来选/组合/适配候选。分工要点：DataTree 维护分支局部的搜索结构，Global Memory 存节点级结果，而 Data Pool 把发现的数据源做成跨分支可复用——三者各司其职、避免把「发现的数据源」与「搜索记忆」混为一谈。

有限预算下的树生长与调度

由于每个候选状态都可能要跑一次下游训练评测，预算有限时无法枚举 $\Pi_B(D_0)$。DataMaster 用两个预算控制机制：

  • Growth policy（生长策略）： 决定下一步是开一个 red node（拓宽候选空间）还是开一批 black node（深挖利用）。树的生长是动态自适应的：red node 的输出只是「潜在数据集」，其是否真有用必须等被选中、适配、由 black node 测过才知道。因此每个 red node 后面跟一小批并行的 black node，探索对新候选的不同利用假设。某个 black node 评完后，由一个轻量外部 controller 根据最新训练反馈与当前分支上下文，决定该分支是继续 black-node 精修、还是开新 red node 做更广探索——当前配置还有提升空间就「深挖」，分支覆盖不足或候选基本用尽就「拓宽」。
  • Scheduling policy（调度策略）： 在待执行节点间分配下一单位评测预算。采用贪心调度，选「估计最有助于逼近高分数据状态」的下一个未执行节点。下一执行节点 $v_{\text{next}}=\arg\max_{v\in\mathcal{N}}\mathrm{Score}(v)$（$\mathcal{N}$ 为已生成但未执行的 frontier 节点），用一个 Upper Confidence Bound（UCB1）风格的规则平衡分支质量与探索不足：

$$ \mathrm{Score}(v) = \frac{R_v}{N_v} + c_t\sqrt{\frac{\log N_{\mathrm{Par}(v)}}{N_v}}, $$

其中 $R_v$、$N_v$ 是以 $v$ 为根的分支的累积 reward 与访问次数，$N_{\mathrm{Par}(v)}$ 是父节点对应计数，$c_t$ 是随时间衰减的探索系数。black node 的评测 score 会向上 backpropagate 更新分支统计，使反复成功的分支更容易被优先，同时仍保留对探索不足方向的激励。

实现要点（多取自 Appendix C，对复现重要）
  • 即时 reward 设计：$r_v=\epsilon$（red node 无错完成，$\epsilon=0.01$ 小正常数，保证数据发现步拿到非零信用但不盖过真实训练信号）；$r_v=y_v$（black node 的评测 score）；$r_v=0$（buggy/失败节点）。
  • Backpropagation：节点 $v$ 完成后，沿从 $v$ 到 root 的整条路径 $\mathrm{Path}(v)$ 更新：$N_u\!\leftarrow\!N_u+1$、$R_u\!\leftarrow\!R_u+r_v$，使父/祖父节点累积对其子树质量的证据，从而在不同深度的分支间公平比较。
  • 新节点初始化：$N_v=0,R_v=0$；为避免 UCB1 除零，未访问节点 $\mathrm{Score}=+\infty$（乐观初始化），保证每个新 frontier 节点都先被执行一次，再回到标准 UCB1。
  • 探索系数衰减：$c_t$ 三段分段线性——探索期（$tt_2$）钳到 $c_{\min}$。
  • 默认超参（Table 10）：$c_0\!=\!1.414(\approx\!\sqrt2)$、$c_{\min}\!=\!0.5$、$\alpha\!=\!0.01$、相位比 $(p_1,p_2)\!=\!(0.3,0.7)$、$\epsilon\!=\!0.01$、最大预算 $T\!=\!40$ 轮；树结构 num_red=1、初始 num_black=5、max_black_per_red=5。
  • 硬件：MLE-Bench Lite 在 8×NVIDIA H20 上跑，每 4 个任务分配 1 张 H20 以支持并行；主结果统一用 GLM-5 作 backbone、12 小时时限、各任务相同训练算力预算。

④实验评估 Evaluation

评测设置

  • Benchmark： 两个互补设定，差别在「是否提供初始数据」。
    • MLE-Bench Lite（ref [8] 的 held-out 子集，22 个 Kaggle 竞赛，跨 tabular / vision / NLP / time-series）：每个任务给定固定竞赛数据 + 初始代码解，agent 只能在数据侧操作（选子集、清洗、特征工程、调格式），模型训练代码不变。考察「在已有数据上做数据优化」。
    • PostTrainBench（ref [29]，7 项能力）：衡量 agent 在只给算力与时间预算、不给任何训练数据的情况下能多好地 post-train 一个 base LM——必须从零发现并 curate 数据。7 项能力为：AIME 2025（数学推理）、Arena-Hard Writing（指令跟随/创作，LLM-as-judge）、BFCL（function calling）、GPQA（研究生级科学知识）、GSM8K（小学算术推理）、HealthBench Easy（医疗问答）、HumanEval（代码生成）。考察「开放世界数据发现」。
  • 对比基线（四类）：(i) Frontier agents——Claude Code CLI、Codex CLI；(ii) Dataset Search Agent——DatasetResearch（纯数据搜索，发现后用 Claude Code 适配，并控制总数据量与 DataMaster best node 对齐）；(iii) End-to-End ML Agent——ML-Master 2.0（可同时改数据与代码）；(iv) Data-Centric Training Framework——DataFlex（建于 LLaMA-Factory，只做训练时的数据选择/混合比/重加权，不做自主数据发现；收到与 DataMaster best node 相同的 curated 数据集，用以隔离「训练时优化」与「数据发现」）。
  • 参照点（Reference Points）：
    • Initial Score（初始解）：MLE-Bench Lite 上 = 把 ML-Master 2.0 的代码去掉数据模块后跑出的成绩（隔离固定训练流水线本身的贡献），medal/gold = 35.91% / 22.73%；PostTrainBench 上 = base model Qwen3-1.7B-Base 不做 fine-tune。
    • Human Score（人类专家）：PostTrainBench 上 = Qwen3-1.7B 的官方 instruct 版（阿里专家团队用精心 curate 的数据训出），作为「专家级 post-training」的强参照，平均分 46.91%。
  • 指标（方向）： MLE-Bench Lite 四指标——Medal Rate（达到 Kaggle 奖牌线的竞赛比例，↑）、Gold Medal Rate（限金牌，↑）、Avg. Overcome Rate（DataTree 中提交分数超过初始解的节点比例，↑）、Performance Gain（DataTree best 相对初始解的平均相对提升，↑）；PostTrainBench——每项能力的准确率 + 宏平均 Avg（↑）。除特别标注外，所有 agentic 系统都用 GLM-5 驱动（Codex 用 GPT-5.2-Codex）。

成本结构与数据泄漏审计

成本（原文 Table 4，641 red / 340 black 节点）：black node 更贵——用 3.0× 的 API 预算、2.4× 的时间（数据选择/处理/构造 DataLoader/下游训练评测）；red node 主导搜索广度，执行次数近 black node 两倍，体现开放式外部发现的特性。整体上 DataMaster 比基线更「重」（原文 Table 7：平均每任务输入 2690.3K / 输出 596.3K token、API 成本 \$1.61、用时 718.7 min），换来明显更高的 medal/gold——属于「用更多搜索预算换更强数据配置」的取舍。

作者用多层协议防泄漏：① source filtering（屏蔽 benchmark 页面、竞赛讨论、含 test-split 名的 URL）；② hash-based dedup（存 test-set hash，过滤精确匹配的训练样本）；③ provenance tracking（对每个外部数据点记 URL、时间戳、内容 hash 以便审计）。这一审计回应了「自主搜外部数据是否会作弊」的核心质疑。

⑤洞见 Insights

Insight
  • 「只动数据、不动代码」是一条被低估的优化杠杆。 当模型/训练配方日益标准化，把 agent 的整个 action space 限定在数据状态上（发现、选择、清洗、转换、复用），仍能在经典 ML 工程与 LLM post-training 两端都大幅提升强初始解，甚至在 GPQA 上超过人类专家 curate 的数据。这把「data-centric AI」从「人手工调数据」推进到「agent 自主在数据侧做闭环搜索」。
  • 开放式数据工程天然适配「树 + 共享记忆」。 数据迭代是可分支、路径依赖、累积、且只能由下游训练验证的——用 DataTree 组织搜索、Data Pool 跨分支共享候选、Global Memory 累积复用，把零散数据操作变成有结构的搜索。消融显示 Memory 是收益的命门（去掉它几乎抹平全部增益），印证了「累积记忆」对 long-horizon 数据迭代的决定性作用。
  • 数据 curation 的质量 > 训练时调度。 在收到同样 curated 数据集的前提下，只做训练时优化的 DataFlex 平均落后 DataMaster 12.08pt——说明「发现并构造对的数据」比「在给定数据上调混合比/重加权」更值钱，两者正交、原则上可叠加。另外，backbone 的 tool-use 能力（每节点 tool call 数）直接限制了 long-horizon 数据搜索的上限（GPT-5.4 的反例）。

局限与未来

  • 成本偏高：DataMaster 的 token/时间/API 开销明显高于轻量基线（每任务约 \$1.61、12 小时上限、近 3M token），black node 的下游训练评测尤其贵；在更紧的预算或更大规模任务下的性价比仍需观察。
  • 泄漏风险需持续治理：开放世界搜数据天然带 contamination 风险，本文用三层协议 + n-gram 审计缓解并报告了零精确匹配，但这是工程性防护、依赖 blocklist/hash 的覆盖度，换任务/换数据源时需要重新审计。
  • 开源不完整：截至解读日，仓库只放出 MLE-Bench workflow，PostTrainBench 部分标注「Coming soon」，权重与 agent 自建的数据集均未发布，复现 PostTrainBench 结果暂受限。
  • 评测面有限：主结果集中在 GLM-5 backbone、MLE-Bench Lite（22 个 Kaggle 任务）与 PostTrainBench（7 项、Qwen3-1.7B-Base）；更大模型、更多模态/任务上的泛化是自然的后续。作者在结论中也呼吁未来自主 ML 系统应更重视「数据如何被发现、选择、转换、验证与跨任务复用」这一数据侧维度。

与相关工作的关系（基于本文自身论述）

与 ML engineering agents（AIDE、SELA、MLZero、MLE-STAR、ML-Master）相比：它们把数据当固定给定、只在模型代码/架构/超参上做树搜索；DataMaster 反过来——代码固定，整个 action space 作用在数据状态上，并以下游训练表现为唯一优化信号。PostTrainBench 朝「数据感知」迈了一步（允许 post-training 时 curate 数据），但数据仍是辅助 action 而非中心目标。与 data-centric agents（CAAFE 做 feature engineering、Co-STEER 的全 data-centric 开发环、Dataforge 双反馈环清洗 tabular、LAMBDA 迭代代码生成、DatawiseAgent 编排 notebook 工作流）相比：这些工作里数据源都「事先给定」，agent 只处理/变换给定数据集，而 Spider2-V、KramaBench 等 benchmark 也揭示这类 agent 在复杂异构真实流水线上仍吃力。DataMaster 的根本不同在于源空间本身是开放式的——agent 必须从零发现、评估、适配外部数据，并把下游训练表现作为引导这一过程的唯一质量信号；它面向「模型代码固定、整个 action 都在数据侧」这一互补 regime，并跨迭代累积记忆。
