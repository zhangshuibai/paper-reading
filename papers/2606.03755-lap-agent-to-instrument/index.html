TL;DR — 自主科学（autonomous science）正从「demo」走向「基础设施」：LLM agent 会设计实验，self-driving lab 会执行实验，但每个系统都得从零手搓「会推理的 agent」到「物理仪器」之间那根线。作者把 agentic 生态抽象成三条边：MCP 管 agent↔tool、A2A 管 agent↔agent，而谁都没管 agent↔instrument 这第三条边。LAP（Lab Agent Protocol）就是来填这条边的协议设计。它沿用 A2A 的 peer-to-peer / 先发现后委派 / task 生命周期骨架，再补上四个「物理世界原语」：InstrumentCard（签名的能力 + 物理上限描述）、reservation（一等公民的独占租约/上锁）、safety-fence 握手（把人类授权 token 用密码学绑死到「这一个 task + 这组参数」上，给危险/不可逆操作上闸）、以及 MeasurementResult（强制带单位 QUDT/UCUM、带标定、带不确定度、带可复现 provenance 的结果 schema）。LAP 不替代 SiLA 2 / OPC-UA，而是把它们封装在仪器适配层之下；它和 A2A/MCP 传输兼容。诚实地说：这是一篇 v0.1 设计规范，自承「尚无大规模实现、无实证评估」，参考实现只是计划中的下一步。

★领域背景速补（读者只懂 LLM/agent，不懂实验室自动化）

这篇论文落在「AI agent × 真实实验室仪器」的交叉地带。下面把第一次出现的领域专名讲清「是什么 / 为什么难 / 和本篇什么关系」，术语保留英文。

autonomous science / self-driving lab（SDL，自主科学 / 自动驾驶实验室）

把「提出假设 → 设计实验 → 真去做实验 → 读结果 → 据此改下一步」这个科研闭环交给 AI 自己跑、尽量无人值守，就叫 autonomous science；承载它的物理实验室就是 self-driving lab。可以类比成「科研版的 agentic loop」：只不过 loop 里的「动作」不是调一个软件 API，而是真的让一台机器去合成化合物、扫一张衍射图。代表性 demo：Berkeley 的 A-Lab（17 天无人值守合成出 36/57 个目标无机材料）、GPT-4 驱动的 Coscientist（自己规划并在云实验室跑钯催化偶联反应）、ChemCrow（给 GPT-4 接 18 个化学工具）、Liverpool 的「移动机器人化学家」（8 天做 ~700 个光催化实验）。本篇的出发点：这些系统的「推理层」越来越强，但「推理 agent ↔ 物理仪器」那根线全是一次性手搓的。

agent-to-instrument：为什么「让 LLM 控制真实仪器」比调软件 API 难得多

一次 MCP 工具调用是无状态、瞬时、可重试、出错没代价的（调错了再调一次）。但一台物理仪器是另一回事：

  • 有状态、独占（stateful / exclusive）：一台粉末衍射仪同一时刻只能跑一个实验，两个 agent 不能交错操作；一只移液机械臂不能同时在两个位置。
  • 慢且有真实代价：一次操作几分钟到几小时，会消耗真实试剂、用掉不可替代的样品、烧掉机时和钱。
  • 安全攸关、操作不可逆（safety-critical / irreversible）：激光、高压、强辐射、靠近人的机器人——错了会伤人或毁设备，且很多动作（退火、销毁样品）做了就回不去。
  • 结果要带物理含义：仪器吐出的「25」毫无意义，必须配上单位（unit）、标定参照（calibration）、不确定度（uncertainty）才是一次「测量」。

这四点，MCP 的 tool schema 和 A2A 的 AgentCard 里一个字段都表达不了。再加上：agent 是「概率性、目标驱动」的客户端（从一句自然语言目标出发去推理），而既有实验室标准全都假设客户端是「确定性软件」（事先就知道该发哪条命令）。这就是本篇要解决的核心鸿沟。

既有「实验室互联标准」一览（LAP 要和它们划清关系）

SiLA 2：实验室仪器连接里最成熟的开放标准，基于 gRPC/HTTP-2 + Protobuf，用 FDL（Feature Definition Language）给仪器命令做机器可读的类型化描述，还自带 mDNS 即插即用发现。强在「确定性编排」，但 schema 是设计期静态的，没有运行时能力协商、没有每能力的安全危险分级、没有独占预约、结果里没有结构化不确定度，规范里也完全没有 AI/LLM 的位置。
OPC-UA + LADS：工业级、安全的信息模型；LADS 这个 Companion Spec 给它加了实验室概念（FunctionalUnit、ResultType、设备状态机）。但节点图很重、模型很「通用」、本身没有领域语义，且是模板式的，不能在运行时组合出新流程。
AnIML / Allotrope(ADF)：都是「数据静止层」标准——表示已经测完的分析化学结果（AnIML 是 XML、Allotrope 是 HDF5 + 庞大 OWL 本体）。语义很丰富、审计/合规很强，但都不能命令仪器、不能流式实时数据，管不到控制回路。
LIMS / ELN：实验室信息管理系统 / 电子实验记录本，是「记录与管理」层（样品台账、流程、结果归档），不是 agent 实时驱动仪器的 wire protocol。（本篇主要把它们归入数据/管理层，重点对比的是 SiLA 2 / OPC-UA / WoT TD。）

还有两个本篇反复引用的「积木」：W3C WoT Thing Description (TD) 1.1——一个用 JSON-LD 描述 IoT 设备 Property/Action/Event、自带 unit 注解（推荐用 QUDT）的 W3C 推荐标准，LAP 的 InstrumentCard 明说自己是它的一个「profile（扩展剖面）」；以及 QUDT / UCUM——分别是「物理量种类的本体 URI」和「单位编码标准」，LAP 用它们强制给每个数值打上「这是摄氏度还是千帕还是计数」的标签。

通用 agent 协议：MCP 与 A2A（LAP 的直接参照系）

MCP（Model Context Protocol，Anthropic，2024-11）是「竖直」的 agent↔tool 协议：一个 LLM（Host）通过 Client 连到 Server，Server 暴露 Tools（带 JSON Schema 的可调函数）、Resources、Prompts；它是 hub-and-spoke，两个 Server 之间不能直接通信，也没有 task 委派或资源预约概念。A2A（Agent2Agent，Google，2025-04）是「水平」的 agent↔agent 协议：每个 A2A server 在 /.well-known/agent-card.json 发布 AgentCard 声明自己的 skills，agent 间用 JSON-RPC 2.0 + 一个八状态 Task 生命周期（submitted / working / input_required / auth_required / completed / failed / canceled / rejected）协作。二者都已捐给 Linux Foundation 中立治理。一句话记忆：MCP 是「工具连接的 USB-C」，A2A 是「agent 协作的 HTTP」。

LAP 的立论就建立在这张图上：MCP 和 A2A 已经标准化了三角形的两条边，LAP 来补第三条边——而且它选择站在 A2A 这条 peer-to-peer 线上（每台仪器是一等公民 agent，自己发布签名的 InstrumentCard），而不是像下文的 SCP 那样站在 MCP 的 hub-mediated 线上。

怎么衡量一个 agent-to-instrument 协议好不好？

本篇没有跑实验，所以没有量化指标。但从它的论证框架可以读出「一个这类协议应该覆盖哪些维度」：能否运行时发现仪器能干什么（覆盖异构设备：SiLA 2 / SCPI / 自制机器人都能用一张卡描述）、物理量类型是否强制、独占预约是否一等公民、安全分级 + 人类授权是否在协议层而非 UI 层、结果是否带标定/不确定度可复现、能否跨实验室联邦、以及是否有运行实现。最后这一项 LAP 自己填的是「none（仅设计）」——这正是它最大的短板，下文会反复强调。

②研究动机与贡献 Motivation & Contribution

背景与问题：自主科学的瓶颈不在「智能」，在「那条边」

作者的核心观察很犀利：autonomous science 受限的不是 agent 的智能，而是「智能与物质相遇」那条边上缺一个协议。过去十年的 demo（A-Lab、Coscientist、ChemCrow、Liverpool）都证明了「AI 自主做实验」物理上可行，但它们的「推理 agent ↔ 物理仪器」那根线无一例外是一次性手搓的：

  • A-Lab 的编排框架 AlabOS 维护 28 个设备实例、16 种设备类型，每一种都要手写适配器（MODBUS、TCP/IP、串口、HTTP、XML-RPC）；
  • Coscientist 对每台新仪器都得在运行时去解析厂商 API 文档，扩展性差、还看文档质量；
  • ChemCrow 的 18 个工具个个是定制集成，要扩到上百台仪器就需要一个标准；
  • 多份 review（含 NIST 的专项计划）都把症结说穿：SDL 各自为政、仪器 API 五花八门、命令/遥测/安全语义都不统一，集成靠「fragile hacks（脆弱的拼凑）而非健壮接口」。

这本质是经典的 $O(n \times m)$ 集成问题（$n$ 个 agent 对 $m$ 台仪器，每对都要单独打通），而一个协议的意义恰恰是把它压成 $O(n + m)$。软件世界已经在一年内为这类问题收敛出了协议——但只针对软件，没针对「物质」。

本文的切入点：三条边，两个协议，缺第三条

作者把 agentic 科学生态组织成三条通信边（见下方图 1）：MCP 标准化了 agent-to-tool（竖直，向下够工具），A2A 标准化了 agent-to-agent（水平，agent 互为 peer），而 agent-to-instrument 这条边「不能被任一邻居还原」——它既不像「无状态瞬时的工具调用」，也不像「本身会推理的 peer agent」，而是一个有状态、被独占、安全攸关、以物理单位度量、操作缓慢且有后果的物理资源。从这四个属性，作者推导出任何 agent-to-instrument 协议都必须提供的四个原语（P1–P4），这成了整篇报告的组织主线。

核心贡献

  1. 问题框定（三条边 → 四原语）：把「仪器控制」定位为 agentic 生态的第三条边，并从其物理属性推导出协议必须提供的四个 primitive：(P1) 能力与物理上限描述、(P2) 独占预约、(P3) 安全协商、(P4) 物理上诚实的结果。
  2. InstrumentCard（P1）：一个签名的 JSON-LD 能力卡，是 A2A AgentCard 和 W3C WoT TD 1.1 的扩展，额外带 instrument class、每能力的 safetyClass、reversible、物理上限、标定状态、和给自然语言落地用的 intentTags。
  3. 完整的交互层：一套 JSON-RPC 2.0 方法集；把 A2A 八状态扩成带物理态（safety-hold / paused-fault / sample-wait）的 task 状态机；一等公民的 reservation 租约；以及一套结构化、可让 agent 自动重规划的错误模型。
  4. safety-fence 握手（P3）：建立在 S0–S3 四级安全分类之上，用「operator-confirmation token」把人类授权用密码学绑死到「这一个 task + 这组参数的哈希」上——让「autonomous agent 操作危险硬件」从一句开放陈述，变成一个有界、可审计、单次可用的陈述。
  5. MeasurementResult schema（P4）：强制每个结果都「物理类型化（QUDT/UCUM）、锚定标定、携带不确定度、附可复现 provenance 清单 + 签名」，让结果天然 FAIR、可复现、可机器验证。
  6. 发现与跨实验室联邦设计：按能力跨实验室检索、范围受限的凭证授予、物理样品的 chain-of-custody，并用一个端到端的闭环 campaign 把所有原语串起来演示。
一句话定位 LAP 不是新传输、不是新数据格式。它复用 JSON-RPC、JSON Schema、QUDT/UCUM、WoT TD、A2A 的发现与 task 模式；它封装（而非取代）SiLA 2 这类设备标准——后者活在 LAP 协议栈的 L0、藏在 Instrument Agent 内部。它的新意只在于：把物理世界要求的四个原语组合进一个协议，这是此前任何 agent 协议都没提供过的。

③方法：协议架构 Method

下面按「整体（三条边 / 五角色 / 六层）→ 四原语逐个展开 → 一次闭环时序」来讲。所有图均为依据原文重绘的内联 SVG。

五个角色与最小拓扑

LAP 定义五个角色（见图 2）。记住一句：最小可用拓扑就是「一个 RA 直连一个 IA」，LC/FR 是随实验室长大和联邦化才加进来的——这刻意模仿了 web 从孤立服务器长成联邦网络的路径。

  • Research Agent (RA)：自主科学家本体。提假设、规划 campaign、拆成仪器 task、解读结果、驱动闭环。它说 LAP 的客户端。
  • Instrument Agent (IA)：包住恰好一台物理仪器（或一个逻辑集群）。它把 LAP task 翻译成下面的厂商 SDK / SCPI / 串口 / OPC-UA / SiLA 2 调用，独占持有仪器的状态、安全 interlock、标定和预约表；发布 InstrumentCard，说 LAP 的服务端。
  • Lab Coordinator (LC)：每实验室一个 broker。单台仪器时可选，多仪器且有跨仪器危险的工作流时必需。它聚合 InstrumentCard、仲裁预约争用、执行全实验室安全策略、在仪器间中转样品 chain-of-custody——也是唯一能看见「整条工作流」的角色，故是 (C) 类「工作流级安全」的执行点。
  • Federation Registry (FR)：可选的跨实验室目录。按能力/位置/可用性/访问策略索引仪器，返回签名指针；永不代理控制。
  • Human Operator / Safety Authority (HO)：不是 agent，而是一个信任锚：签发/转交 operator-confirmation token 来给安全围栏的操作放行，并能下达急停。

六层架构：LAP 只规范中间三层半

LAP 分六层（图 3）。关键设计是刻意把 L0 和 L5 留在范围之外：L0（设备抽象，藏在 IA 内部）「封装它本身就是协议的全部目的——好让 RA 永远看不见厂商 SDK」；L5（科学本身，藏在 RA 内部）「LAP 给钩子但不替你做主」。正是这条边界，让同一张 InstrumentCard 能把 SiLA 2 仪器、SCPI 示波器、自制机器人统一成一个 agentic 接口暴露出来。

四个物理世界原语（逐个展开）

P1 · InstrumentCard：仪器的「签名能力卡」

它是 LAP 的中心 artifact，相当于「仪器版的 A2A AgentCard」，实现为 W3C WoT TD 的一个 profile，服务在 /.well-known/instrument-card.json 并由实验室权威签名。除了与 AgentCard 共享的身份/传输/安全字段外，它补上了 agent 真正需要的仪器结构：取自能力本体的 instrumentClass；一组 capabilities，每个带物理类型化的输入/输出 schema、safetyClass、reversible 标志、estimatedDuration、physicalLimits 与 interlocks、以及用于落地自然语言的 intentTags；连续遥测 streams；和一个 calibration 块。

方法集与「自然语言只在边缘出现」的关键决定

所有交互都是 JSON-RPC 2.0；流式方法升级为 SSE，长任务可再注册 webhook。方法按它服务的原语分组：instrument.describe / getState（发现）、reservation.request / renew / release（P2）、task.submit / get / stream / cancel / subscribePush（每个改变物理状态的动作都是一个可审计/可取消/可流式的 Task）、safety.requestAuthorization / provideToken / emergencyStop（P3）、intent.resolve（唯一吃自然语言的方法）、registry.query / advertise（联邦）。

核心设计：自然语言永不直接驱动硬件 intent.resolve 把一句自然语言目标（「从 10° 到 80° 做一次细步长粉末扫描」）映射成一个提议（proposal）——一个填好、类型化、过了上限检查的结构化 task.submit 载荷——RA 必须显式确认才能提交。所以硬件永远只通过「经过校验的结构化调用」被驱动，绝不直接来自模型输出。作者强调这主要是「可靠性 reliability」决定而非安全保证：它把 LLM 的随机性关进一个可检视、可重放的提议里；真正的安全门是其后那个确定性的 task.submit 校验（schema + physicalLimits 检查，S2/S3 还要 operator-token 握手）。intent.resolve 是可选的、best-effort 的，提议不携带任何特权信任。

P2 · Reservation：把「互斥」从约定变成协议原语

因为仪器一次只能服务一个实验，LAP 把互斥做成显式的、带租期的原语。reservation.request 返回一个 reservationToken 和一个 time-boxed 租约；在独占仪器上 task.submit 必须带有效 token，租约必须用 reservation.renew 续命否则到期自动释放——这防止一个崩溃的 agent 把仪器永久死锁。两种模式：exclusive（独占提交状态变更 task）与 shared-read（多个并发只读）。单调递增的 epoch 让「在已失效又重发的租约下提交的 task」被检测并以 LeaseExpired 拒绝。两个操作无条件绕过预约：instrument.describe（发现永远允许）和 S0 快路径的 safety.emergencyStop（急停连凭证范围都绕过——「halting 一台失控仪器，绝不能被锁或被授权范围不足挡住」）。租约还可带 budgetCap（货币 + 金额），把整条自主 loop 的累计 operationalCost 封顶，超了就报 BudgetExceeded——这正是为了挡住「每步都合规、但 loop 整体吃掉无限机时/试剂/账单」这种 (C) 类失败。

Task 状态机：在 A2A 八态上加三个「只有物理过程才有」的状态

提交后，参数先对能力 schema 与 physicalLimits 校验；IA 还必须在标定过期（calibration.validUntil 已过）时拒绝该 task——因为过期标定下跑出的结果「物理上有效但科学上不可信」（(C) 类完整性），这类失败给 rejected（标定情形报 CalibrationExpired）。有效 task 经 queued 到 running。三个物理态从正常路径分叉：safety-hold（S2/S3 缺有效 operator token、授权握手进行中）、paused-fault（硬件 interlock 跳了，可自动恢复到 running 或终结为 failed）、sample-wait（等另一台 IA 经 chain-of-custody 送来物理样品）。成功终结为 completed 并携带一或多个 MeasurementResult；safety.emergencyStop 把任何非终结态打到 failed 并带 eStop 标志。

P3 · safety-fence：把人类授权用密码学绑死到「这一个动作」

LAP 把安全分成三类，各放在能执行它的那一层：(A) 设备物理安全（不越物理包络、守 interlock、出错即停）——局部且瞬时，整体下放给 IA（协议只下「MUST 拒绝越界 task」的规范义务，把机制当黑盒）；(B) 授权与问责安全（为危险/不可逆/昂贵动作取得并记录人类权威的同意）——跨越人-agent 信任边界、必须跨厂商统一可验证，故在协议层标准化为 safety fence；(C) 工作流/完整性安全（单 IA 看不见的组合危险、过期标定结果、资源/成本失控）——交给 LC + 几个协议级声明（sampleCondition、强制标定过期拒绝、budgetCap）。

每个能力声明一个 safetyClass：S0 急停/元操作（永远允许、走快路径）；S1 常规可逆无害（无需围栏）；S2 消耗不可替代样品 / 昂贵 / 难逆（需新鲜 operator token 或常驻策略授权）；S3 危险操作（激光、高压、自燃物、高压、辐射、靠近人的机器人）——需 Safety Authority 签名的 token 且 getState 里正向的 interlock 证明，绝不能由 RA 单方自授。operator-confirmation token 是一个 JWS，其 claims 把这次批准绑到唯一一个动作上：

// operator-confirmation token（原文，节选）
{ "jti":"<nonce>",        // 一次性随机数：token 单次可用
  "sub":"<task-uri>",    "instr":"<instrument-uri>", "cap":"<capability-id>",
  "paramsHash":"<sha256>",  // 规范化参数的哈希（已并入 instr/cap，防跨仪器/能力重放）
  "authority":"<did>", "iat":"<ts>", "exp":"<ts>", "scope":["..."] }

paramsHash 是「单位归一化值 + 字典序排序键」的规范化参数的 SHA-256，且并入了 instr/cap。因此：参数升级攻击（拿温和操作的批准去跑更激进的）会改变绑定摘要从而使 token 失效；跨仪器/跨能力重放被同样的绑定挡住；有效期内的重放被 jti 一次性 + IA 的已兑换登记表挡住——一次人类批准 = 恰好一次执行。这正是让「autonomous agent 操作危险硬件」成为一个有界、可审计陈述的机制。

P4 · MeasurementResult：物理上诚实、可复现的结果

完成的测量 task 返回一个 MeasurementResult。LAP 把四样「即兴结果载荷常常省略」的东西设为强制：每个数值带 UCUM unit、结果带 QUDT quantityKind；calibrationRef 把结果链到它产出时所依据的标定；声明一个 uncertainty 模型；一个 provenance 块记录确切参数（含哈希）、时序、固件、环境、所用 operator token。此外，当产出方 IA 知道一些影响下游操作安全的样品状态（如「这个合成物对空气敏感」「有热历史」），就记进 sampleCondition 块，下一台 IA 必须拿它对自己的 physicalLimits 核对——这正是 (C) 类组合危险检查的协议载体。整个对象被签名。于是结果「by construction 就是 FAIR 且可复现」的：一个 agent（哪怕来自远端实验室）无需带外知识就能解读它，并能验证「谁、在什么条件下」产出了它。

错误模型：让 agent 能自己重规划

LAP 在 JSON-RPC 的 -33xxx 段保留了一批物理世界错误码，刻意紧贴 JSON-RPC 自身保留段之下。关键是每个错误都「对 agent 可执行」：ParamOutOfPhysicalLimit (-33010) 回显越界参数及其上限，让 agent 重规划；SafetyAuthorizationRequired (-33020) 直接带上授权挑战；ReservationConflict (-33002) 报当前租约持有者与到期时间。其余如 LeaseExpired、SafetyTokenInvalid、InterlockTripped、CalibrationExpired、SampleNotPresent、ChainOfCustodyBroken、BudgetExceeded 等，都按「返回结构化上下文供 agent 无需人类介入即可重规划」来设计。

联邦：从单台仪器到「Scientific Internet of Instruments」

发现是分层的：实验室内单台 IA 在自己的 /.well-known/instrument-card.json 被找到，LC 在 /.well-known/lab-directory.json 列出全部仪器。跨实验室时，LC 用 registry.advertise 向 FR 发能力摘要（仪器类、关键能力与 intent tags、当前可用性、访问策略、地理位置）但不发控制端点。RA 发 registry.query（「哪些仪器能对 <50 mg 样品做 xrd:PowderDiffraction、本周可用、对外开放？」）拿到签名指针，再与 owning LC 协商拿到一个范围受限、time-boxed 的凭证，然后直接和远端 IA 说 LAP。registry 永远只是目录、绝不代理控制——这既压低时延，又让每个实验室对自己的仪器保有唯一权威；一个被攻陷的 registry 无法伪造能力或把控制重定向到冒名者（每张取回的 InstrumentCard 都先验签）。物理样品在仪器/实验室间移动，由 sample URI 上的 chain-of-custody 追踪（LAP 追踪但不亲自强制）。

一次闭环 campaign（端到端把原语串起来）

作者用一个材料 RA「为稳定性优化钙钛矿成分、混用本地+远端仪器+一个危险步骤」的场景把所有原语串成一圈（图 6 给出本地部分时序）：① registry.query 找到本地合成机器人 + XRD + 远端 PL 光谱仪；② reservation.request 拿合成机器人与 XRD 的两小时独占租约；③ intent.resolve（「合成 Cs₀.₁₇FA₀.₈₃PbI₃，100°C 退火 10 min」）+ 确认 task.submit，合成 task 跑到 completed，产物是一个 sample URI；④ LC 把样品路由到 XRD（custody 交接），XRD task 在样品到位后离开 sample-wait；⑤ 提交 S1 的 powder-scan 并开 task.stream 看实时图样，完成得到签名+类型化的 MeasurementResult；⑥ RA 分析相纯度，决定还要测 PL，联邦化访问远端光谱仪、寄送样品、远程重复测量回路；⑦ 某个精炼步骤需要一次激光退火（S3）——IA 把 task 移入 safety-hold 并发 safety.requestAuthorization，人类批准一次、token 绑死到那一个 task+参数、退火执行、loop 继续；⑧ RA 更新代理模型、提议下一个成分。每一步都可经 task provenance 审计。

实现要点 / 设计取舍 传输选 JSON-RPC 2.0 over HTTPS+SSE（而非 SiLA 2 的 gRPC）——理由是「生态对齐」而非性能：JSON-RPC 已是所有主流 agent 框架的 wire 格式，一个说 LAP 的 IA 因此能被任何 A2A 兼容的 RA「零额外工具」触达；gRPC 的 protobuf 编译/二进制调试/HTTP-2 防火墙问题对「写 Python 的科学仪器长尾」成本过高。对少数 kHz 级探测器帧流，streaming 子句允许在规范 SSE 之外并存 WebSocket/gRPC 绑定。注意「零额外工具」只适用于 A2A 形态的结构化表面（task.* / reservation.*）；自然语言 intent.resolve 与物理扩展字段（reservation、safety class、physical limits）仍需 LAP-aware 客户端才能解读。

④评估：这是「设计论证」，不是实证 Evaluation

必须先说清楚 本文没有实验、没有 benchmark、没有量化结果、也没有运行实现。论文自承 LAP v0.1「benefits are argued architecturally rather than empirically validated（其收益用架构论证而非实证验证支撑）」。所以这一节不是「成绩单」，而是「作者用什么论据说服你这个设计是对的」，以及它和同类协议的对比表。这是它有别于本仓库其它论文的最大特点——读者切勿把它当成一个已验证的系统。

验证方式：维度对比 + 端到端 walkthrough + 威胁模型

  • 对比基线：SiLA 2、MCP、A2A、SCP（Science Context Protocol，上海 AI Lab，2025-12，目前在 Intern-Discovery 平台集成 1600+ 工具）。
  • 「指标」：不是数值指标，而是一组定性维度——主要边、发现机制、自然语言/intent、物理量类型、预约、安全围栏、标定/不确定度、跨实验室联邦、传输、是否有运行实现。
  • 论证载体：第三节那条端到端钙钛矿闭环 walkthrough（说明四原语可组合），加上一份显式的 security/threat model（说明 safety fence 防住什么、防不住什么）。

和最接近的对手 SCP 的三点结构差异

  1. 去中心化 vs hub-mediated：SCP 沿 MCP 的 hub-mediated 模型向上扩，所有仪器访问都经 SCP Hub 代理、能力协商是「agent↔Hub」，Hub 成为联邦运行的单点信任/单点故障；LAP 沿 A2A 的 peer-to-peer 线，每台仪器是发布自己签名 InstrumentCard 的一等 agent，RA 直接和 IA 协商（LC 是可选编排层、永不是通往仪器的唯一路径）。
  2. 正式的安全围栏：LAP 有 S0–S3 + 绑定 operator token；SCP 没有。
  3. 物理上诚实的结果：LAP 强制类型化/标定/不确定度结果；SCP 把结果内容当不透明对待。

⑤洞见与局限 Insights & Limitations

Insight
  1. 「三条边」是个极简而有力的框架。把 agentic 生态拆成 agent-to-tool（MCP）/ agent-to-agent（A2A）/ agent-to-instrument（LAP），一眼就让人看到「物理世界那条边」为何无法被前两条还原——并由此推导出协议必须有的四原语。这种「从对象的物理属性反推协议原语」的方法论，可迁移到任何「让 agent 驱动真实世界资源」的场景（机器人、能源、医疗设备……）。
  2. 把人类授权「绑死到参数哈希 + 单次可用」，是让「agent 操作危险硬件」从空话变成可审计陈述的关键招。它把模糊的「人在回路」精确化为：agent 的权限恰好等于「人或策略签过名的 (task, 参数) 对集合」，每个只能兑现一次。这个 pattern 对任何「高风险不可逆动作需人类背书」的 agent 系统都有借鉴价值。
  3. 「自然语言只在边缘、且只产出待确认的结构化提议」——把它定位为可靠性而非安全机制，是难得清醒的工程判断。作者反复强调真正的安全门是其后那个确定性校验 + token 握手，而不是 LLM 解析本身。这避免了「因为 LLM 看了一眼就信它」的常见误区。

局限（论文自己诚实列出 + 我的判断）

  1. 规范优先，实证是「下一阶段」——这是最大短板。LAP v0.1 是协议+schema 规范，不是被 benchmark 过的系统；收益靠架构论证而非生产部署/多站点一致性数据/对照用户研究。最小的去风险实验（单机模拟 XRD IA + 最小 Safety Authority + 脚本 RA + pytest 断言「改任一批准参数就让 token 失效」）也只是计划中。坦白说：在它放出参考实现和一致性套件之前，这更像「一份值得讨论的提案」，而非可落地的标准。
  2. 不管硬实时控制。LAP 面向 task 级编排（提交一次 20 分钟扫描、看进度、取签名结果），明确不适合微秒级伺服/PID 回路；超快泵浦探测、扫描探针快反馈这类必须把内环放在 L0 之下、完全在 LAP 范围之外。
  3. safety fence 假设了一个可信的 operator authority。它防住 agent 重用/升级 token，但防不住「恶意或盖橡皮章的 Safety Authority」乱签、也防不住「被攻陷的 IA」干脆无视 token。LAP 提供可审计的人类监督钩子，但替代不了制度性安全治理、物理 interlock 和操作员培训。还有一个隐蔽风险：合法实验室签了一张「低报 safetyClass / 高报 physicalLimits」的 InstrumentCard——签名是真的，验签抓不到；LAP 只能靠「社区按 instrumentClass 定义安全等级下限 + registry 一致性证明」把它从「不可检测」提升到「可审计」，但消除不了。
  4. 能力本体 instrumentClass 尚未定义。v0.1 只举例（如 xrd:PowderDiffractometer），没有正式、版本化的社区本体；两个实验室对等价仪器用了不同字符串，就彼此发现不到。这需要尚未开始的、持续的社区治理。
  5. 物理样品 chain-of-custody 依赖 LAP 管不到的现实物流。它能记数字化交接，但保证不了「样品真到了对的仪器、没贴错标、技师装的就是 URI 声称的那个」——这要靠条码/RFID/机器人/人工流程，全在协议之外。
  6. intent.resolve 的可靠性是 best-effort、依赖 LLM。受底层模型质量与 intentTags 完整度限制，对模糊请求/仪器术语/边缘情形不保证正确反映意图；「RA 必须显式确认提议」只缓解不消除。
  7. 全球尺度的联邦信任与凭证是未解的治理问题。小规模可信伙伴联盟尚可；扩到跨数千机构、跨法域（数据主权、生物安全法规各异）的「scientific internet of instruments」，所需的签发/校验/吊销/审计跨实验室凭证的治理基础设施不存在，且非 LAP 单方能解。

采纳路径（作者怎么设想落地）

LAP 设计成可增量采纳：一台现有仪器通过一个薄软件适配器把 LAP JSON-RPC 映射到设备原生层（SiLA 2 gRPC server / 厂商 SDK / 裸 SCPI），无需改动仪器或驱动就变成 IA。两阶段：近期由集成方/开源社区把现有仪器包成薄 IA（无需厂商配合）；远期由厂商原生出厂自带 IA（作为产品特性，暴露统一 LAP 面而不泄露专有固件），把适配成本「在厂商处摊一次」而非「在每个部署处重复」——正如设备类标准对 USB 外设、IPP 对网络打印机所做的。最小去风险实验仍是那套单机参考实现 + 一致性套件，之后是社区共建的 InstrumentCard registry（顺带 seed instrumentClass 本体），最终走 MCP/A2A 走过的中立多方治理路径。

与相关工作的关系（基于本文自身论述）

LAP 把自己定位为「peer-to-peer、instrument-as-agent 的 wire protocol」，其能力描述层是 W3C WoT TD 1.1 的 profile（非新贡献）、类型化数据概念建立在 AnIML/Allotrope 惯例上（同样已有），新贡献只在四原语的组合。和邻居们的关系是：与 SiLA 2 / OPC-UA / SCPI 封装（它们在 L0、IA 内部）；与 A2A 组合（一个 IA 就是合法 A2A agent）；与 MCP 共存（MCP 给 agent 计算工具、LAP 给同一 agent 物理仪器，服务正交的边）；与 INTERSECT（ORNL 已部署的联邦架构）的关系「如同 HTTP 之于某个具体服务栈」；并指出 VISION（NSLS-II 首个语音控制 beamtime）独立验证了「confirm before actuate」——正是 LAP 形式化为 intent.resolve → task.submit 两步的那个洞见。
