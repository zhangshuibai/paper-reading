一句话 — AI 集群本质是「一堆会算的芯片 + 把它们喂饱、连通、供电、降温的系统」。本文按从小到大四个尺度拆解模块怎么连：① 封装内——GPU 计算单元紧贴 HBM 高带宽内存（CoWoS 中介层），解决「算得快、喂不饱」的内存墙；② 节点内（约 8 卡，scale-up）——GPU 之间用 NVLink/NVSwitch（铜、超高带宽）黏成「一块大 GPU」；③ 跨机柜（scale-out）——节点经 NIC 网卡→光模块→交换机（InfiniBand 或 RoCE 以太网，spine-leaf 拓扑）连成上万卡，GB200 NVL72 就是「72 卡一个 NVLink 域 + 光网络 scale-out」；④ 存储 & 电力散热——数据从 NVMe/网络存储喂进来，电力（含 800V DC）与液冷包住一切。核心洞见：训练算一步就要等一次通信，所以互联带宽/延迟（而非单卡算力）常常才是集群效率的真瓶颈——这就是「通信墙」。

★领域背景速补（只懂 LLM 也能跟上）

这篇讲硬件，但只要你知道「训练一个大模型 = 反复做前向算输出 → 反向算梯度 → 更新权重，而且要在很多张卡上并行」，下面这些名词就够你跟上了：

  • GPU / die / SM / Tensor Core：GPU 是干「大量并行乘加」的芯片；一颗 GPU 的硅片叫 die；die 里有很多并行计算单元（NVIDIA 叫 SM），其中专做矩阵乘法的小核叫 Tensor Core——大模型 99% 的算力花在矩阵乘上。类比：一个超大车间里几万个只会做乘加的小工人。
  • HBM（High-Bandwidth Memory，高带宽内存）：贴着 GPU 的超快内存，存放权重、激活值、梯度。它和你电脑里的 DDR 内存是「同类不同款」：HBM 把多片内存垂直堆叠、用超宽接口贴近 GPU，带宽数量级约 TB/s。类比：贴身弹药袋，工人伸手就能拿。
  • 带宽 vs 延迟：带宽=单位时间能搬多少数据（车道有多宽）；延迟=一条消息从发到收要多久（路有多远）。AI 集群两者都要——带宽决定能不能喂饱，延迟决定「等通信」要等多久。
  • NVLink / NVSwitch / PCIe：都是「连接线/交换芯片」。NVLink 是 GPU 之间的专用超高速铜互联，NVSwitch 是让一组 GPU 两两全互联的交换芯片；PCIe 是通用得多、慢得多的总线，用来连 CPU、网卡、硬盘。
  • NIC 网卡 / 光模块 / 交换机：NIC（网卡，如 ConnectX/BlueField）是节点「上网」的口；光模块（transceiver）把电信号转成光、塞进光纤；交换机（switch）是网络里的「路口」，把成百上千根线路连起来转发数据。
  • InfiniBand（IB）/ RoCE 以太网：两种主流的集群网络技术。IB 是为高性能计算定制的低延迟网络；RoCE 是「在以太网上跑 RDMA」，让普通以太网也能低延迟直达远端内存。二选一，作用类似。
  • scale-up vs scale-out：本文核心二分法。scale-up（纵向扩展）=在一个小范围内（机柜内）用超高带宽把卡黏成「一块大 GPU」；scale-out（横向扩展）=用网络把很多这样的「大 GPU」连成更大的集群，带宽/延迟相对次之。
  • all-reduce（全规约）：数据并行训练里，每张卡各自算出一份梯度，要把所有卡的梯度加起来再分发回每张卡，让大家权重保持一致。这一步是集群里最重的通信操作，下面动画会重点演示。
  • CoWoS / interposer（硅中介层）：一种先进封装工艺，把 GPU die 和 HBM 放在同一块硅「底板」（interposer）上，用底板里成千上万条超短走线把两者超宽连接。类比：把厨房和冰箱装进同一个吧台，伸手就够到。

◎给小白的心智模型：一支打仗的部队

把整个 AI 集群想成一支需要边打边补给的大部队，「连接」就是不同距离的通信/补给线：

  • HBM（贴身弹药袋）——就挂在士兵（GPU）身上，伸手即取，最快，但装不多。对应封装内的片内总线。
  • NVLink（同屋对讲机）——同一个屋子（机柜）里的战友之间喊话，几乎没延迟、信息量大。对应节点内/机柜内 scale-up。
  • 光网络 / 交换机（跨楼电话总机）——和别的楼的部队联络，得经过电话线（光纤）和总机（交换机），能连到很远很多人，但比同屋喊话慢、也更费劲。对应跨机柜 scale-out。
  • 存储（后方军火库）——弹药（训练数据）的大本营，按需用卡车（NVMe/网络）运到前线。
  • 电力 & 液冷（粮草与空调）——没电没冷却，再强的部队也开不动、会过热罢工。是包住一切的物理约束。
一句话记住 越贴近芯片的连接越快越贵、能连的人越少；越往外越能连远连多、但越慢。设计 AI 集群的全部艺术，就是尽量把要频繁通信的计算放进「快连接」的范围内，少走「慢连接」。

A封装内（package 尺度）：计算 ↔ 内存

最里层的故事：一颗 GPU 怎么和它的内存连。GPU die（含成千上万个 Tensor Core）算得飞快，但它要算的权重和数据得从内存里读。于是把 HBM 高带宽内存直接和 GPU die 封在同一块基板上——靠 CoWoS 硅中介层（interposer）里成千上万条超短、超宽的走线相连，带宽数量级约 TB/s，延迟极低。

为什么这么连：内存墙 memory wall 过去几十年，芯片算力涨得比内存带宽快得多。结果是：Tensor Core 经常算完一批就闲着等数据从内存搬过来——「算得快没用，喂数据的带宽才是瓶颈」，这就是内存墙。HBM 贴着 GPU 封在一起，就是为了把这堵墙推远：用尽可能宽的「内存↔计算」通道，让 Tensor Core 别饿着。这也是为什么 HBM 几乎是当代 AI 芯片的标配，而且谁的 HBM 带宽更高，往往谁的实际算力利用率更高。

方向与依赖：GPU die 依赖 HBM 供数据；HBM 只服务它紧邻的那颗 GPU。两者通过 interposer 的片内总线双向高速读写。注意：这层连接出不了这块封装——它是「最快但最近」的连接。

B服务器节点内（node 尺度，约 8 GPU）：scale-up

一颗 GPU 装不下大模型，于是把约 8 张 GPU 放进一台服务器节点，让它们像一块大 GPU 一样协作。关键连接：

  • GPU ↔ GPU：NVLink / NVSwitch（铜、超高带宽、低延迟）。通过 NVSwitch 交换芯片，节点内每张卡都能高速直达任意另一张卡，组成一个「内存语义」的互联域——意思是一张卡可以像访问自己内存一样去读写另一张卡的 HBM。这就是 scale-up：把多张卡的显存黏成一个大显存池，才装得下、跑得动大模型。
  • GPU ↔ CPU：PCIe。CPU 负责调度、装载数据、跑不适合 GPU 的逻辑。PCIe 是通用总线，带宽远低于 NVLink，所以训练时尽量别让大流量走这里。
  • CPU ↔ DRAM：CPU 自己的大容量内存（DDR），用来缓冲从存储读来的数据、做预处理。
  • 节点 ↔ 外部：NIC 网卡（如 ConnectX/BlueField）。每节点配一张或多张高速网卡，是这台机器通向更大网络的唯一出口——下一节 scale-out 全靠它。
scale-up 的意义 节点内 NVLink 域是整个集群里带宽最高、延迟最低的「远程」连接（仅次于封装内）。所以聪明的做法是：把通信最频繁的并行方式（比如张量并行——一层算子被切到多张卡上协同算）放进 NVLink 域内，让它们「同屋喊话」；把通信没那么密的并行（数据并行）留给更慢的跨柜网络。

C机柜 / 跨机柜（rack → cluster 尺度）：scale-out

要训练前沿大模型，8 张甚至 72 张卡还不够，得把成百上千个节点连成上万卡集群。这一步叫 scale-out（横向扩展），数据流是：

GPU → NIC 网卡 → 光模块（transceiver）→ 交换机（switch）→ …（另一台机器反过来）

  • 介质从铜变光：机柜内短距离可以用铜，但跨机柜距离一长，铜就扛不住高速信号了，于是 NIC 出来的电信号经光模块转成光，走光纤到交换机。这就是为什么光模块/光纤在大集群里数量巨大。
  • 交换机用 InfiniBand 或 RoCE 以太网：两种主流低延迟网络技术，作用类似——让任意两张卡都能（经过几跳）通信。
  • spine-leaf 拓扑：交换机分两层——每个机柜接一台 Leaf（叶）交换机，所有 Leaf 再统一接到上层 Spine（脊）交换机。好处是任意两点之间跳数都一样、且带宽可水平扩展——加机柜只要加 Leaf，不用重排全网。类比：每层楼一个分机房（Leaf），全大楼一个总机房（Spine），打给谁都最多转两次。
scale-up vs scale-out（务必分清） scale-up = 机柜内，铜 / NVLink，超高带宽、超低延迟，但范围小（一个 NVLink 域几十张卡）。
scale-out = 跨机柜，光 / IB / 以太，范围大（上万卡），但带宽/延迟相对次之。
代表型号 GB200 NVL72 就是把两者拼起来的典范：72 张 GPU 用 NVLink 黏成一个超大 scale-up 域（像一台超大 GPU），再用光网络把许多个这样的 NVL72 机柜做 scale-out，连成更大的集群。规模/带宽数字视代际而变，但「域内铜超快、跨域光更广」的结构是稳定的。

D存储 & 物理外壳：数据从哪来、靠什么撑着

  • 存储：NVMe / 网络存储。训练数据（动辄上 PB）放在高速本地盘（NVMe SSD）或专门的网络/并行文件存储里，训练时按 batch 一批批读进 CPU 的 DRAM、再送进 GPU 的 HBM。存储带宽不够，GPU 同样会饿着。
  • 电力（含 800V DC 配电）：一柜动辄上百千瓦，传统机房供电扛不住。新一代机柜趋势是用更高电压的直流配电（如 800V DC）来降低线损、减少铜用量、把电更高效地送到每张卡。这是「能不能开得起这么多卡」的硬约束。
  • 液冷 / 散热：卡越密、功耗越高，风冷就压不住了，于是上液冷（冷板/浸没）。散热是包住一切的物理约束——这里只点一句，详见本站数据中心散热材料那篇。

把 A→D 串起来：数据从存储出发，经 CPU/DRAM 进到 GPU 的 HBM，在 Tensor Core 上算，算出的梯度先在 NVLink 域内聚合、再经 NIC→光→交换机跨柜聚合；而这一切都泡在电力与冷却的物理预算里。下面用动画把这条链走一遍。

▶一步训练的「数据流」：为什么算一步要等通信

把上面的图切到 cluster 视图，点 ▶ 播放数据流，会按下面 7 步逐条点亮关系图的边：

  1. 读 batch：从存储（NVMe/网络存储）读一批训练数据。
  2. 进 HBM：数据经 CPU/DRAM 装入 GPU 的 HBM 高带宽内存。
  3. 前向计算：Tensor Core 逐层做矩阵乘，算出这一批的预测和 loss。
  4. 反向算梯度：从最后一层往回逐层求梯度（每张卡各自算自己那份）。
  5. 梯度 all-reduce（域内）：先在节点/机柜内走 NVLink 把同域各卡的梯度聚合——快，因为是铜、同屋喊话。
  6. 梯度 all-reduce（跨柜）：再跨机柜走 NIC→光模块→交换机，把不同机柜的部分聚合结果合并——这一步走光网络，相对慢。
  7. 优化器更新权重：用聚合后的全局梯度更新每张卡上的权重，回到第 1 步开始下一轮迭代。
关键洞见：communication wall（通信墙） 注意第 5、6 步——所有卡必须等梯度 all-reduce 完成、权重同步好，才能开始下一步计算。也就是说：计算和通信是交替的，算一步就要等一次通信。当卡数从 8 张涨到上万张，通信量和同步等待会迅速变成主要开销——这时候决定集群快慢的，往往不是单卡算力，而是互联的带宽和延迟。这就是「通信墙」，也是为什么超快的 scale-up 互联（NVLink）和高效的 scale-out 网络（IB/RoCE、光模块）如此关键：谁能让「等通信」的时间更短，谁的集群有效算力利用率就更高。
