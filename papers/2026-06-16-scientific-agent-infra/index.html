TL;DR

一句话：两格都是 AI4S 的「卖铲子」基础设施——A 格是让 agent 真去动手操作实验仪器的协议/编排中间件，B 格是给科研 agent 卖可信度量（benchmark / leaderboard / 认证）。两格都很早，但「早」的性质完全不同，对中国创业者的可行性也相反。

Part A 仪器 MCP：nascent（萌芽）——标准林立未统一（SiLA2 / OPC-UA LADS / Allotrope 各占一摊），「真把 MCP 接到物理仪器」的案例屈指可数（LBNL 的 TEM Agent、InstrMCP），业界刚意识到 MCP 对仪器不够用、于是冒出 LAP（arXiv 2606.03755）和上海 AI Lab 的 SCP（Science Context Protocol）两个新协议提案。中国适配结论：可做，且上海 AI Lab 已在卡位——切「自驱实验室编排落地 / 国产仪器 agent 化」，付费方（药企、材料厂、国家实验室）真在中国。

Part B agent 评测：科研专用评测 nascent（几乎全是学术开源、无人商业化）；对比通用 eval 已 emerging→crowded（Scale / Surge / Mercor / LMArena 估值 $10B–$29B 级）。中国适配结论：作为「卖可信度量」的独立第三方生意，科研评测在中国短期跑不通（付费意愿弱、ground-truth 太贵、公信力要官方背书），更宜作为 A 格或模型公司的附属能力，不建议作为独立创业主轴。

哪一格更适合中国早期创业 → A 格（仪器编排 / 自驱实验室）。理由：付费方在中国且预算实在、政策（国产替代 + 国家实验室采购）是助力、可借鉴的国外成功商业模式清晰（Emerald Cloud Lab 的「云实验室订阅 + 按实验计费」、Ginkgo 的「foundry 数据飞轮 + Catalyst/EstiMate agent」）。B 格更适合作为能力嵌进 A 格或模型评测平台，而非单独卖。

① 领域背景速补（读者只有 LLM/agent 基础）

这两格都站在「实验室自动化」和「科研评测」两个对 LLM 圈陌生的领域上。下面把第一次出现的术语讲清「是什么 / 为什么难 / 和本篇关系」，术语保留英文。

A. 让 agent 真去「动手」的世界

  • 自驱实验室 self-driving lab / autonomous lab：一套「机器人 + 仪器 + 调度软件 + 决策算法」的闭环系统，能自己提配方 → 跑实验 → 测结果 → 改配方再跑，无需人盯着。为什么难：现实世界有物理状态（试剂会用完、机械臂会撞、反应不可逆、危险操作要安全确认），不像软件那样可随意重试。和本篇关系：A 格就是给这套闭环装上「LLM/agent 大脑」并标准化它和仪器对话的接口。
  • lab automation / orchestration（实验室编排）：「编排」= 把移液机器人（pipetting robot）、HPLC（高效液相色谱，分离/纯化分子）、plate reader（酶标仪，读微孔板里的光信号）、synthesizer（合成仪）这些异构设备，按一个实验流程协调起来跑。为什么难：每家厂商的设备 SDK/协议都不一样，像 Windows、Mac、打印机各说各话——编排层就是那个「统一翻译 + 调度」的中间件。
  • 移液机器人 / 液体处理（liquid handler）：实验室里最常见的自动化设备，用机械臂+移液枪在微孔板（96/384 孔）之间精确吸/打液体，是生物/化学高通量实验的主力。Opentrons、Hamilton、Tecan、Beckman 是主要厂商。

B. 实验室设备互联标准（这格的「TCP/IP 之争」）

  • SiLA2（Standardization in Lab Automation）：开放、免版税的实验室设备通信标准，2018 年基于 gRPC/Protobuf 重写。把「设备能做什么命令、有什么属性」定义成统一接口。sila-standard.com 🟢
  • OPC-UA LADS（官方全称 Laboratory and Analytical Device Standard，注意常被写作「Agnostic」是早期用法）：把工业界成熟的 OPC-UA 工业通信协议扩展到实验室分析仪器的 companion spec，由 OPC Foundation + 德国 SPECTARIS/VDMA 主导。opcfoundation.org/.../lads 🟢
  • AnIML / Allotrope ADF（数据格式）：实验数据怎么存才能跨厂商/跨实验室读懂。AnIML 是 XML 的开放标准；Allotrope ADF 基于 HDF5+RDF 语义元数据，由大药企联盟推。allotrope.org 🟡。为什么重要：agent 要做决策，必须先能读懂仪器吐出来的数据。
  • LIMS / ELN：LIMS（Laboratory Information Management System，管样品/流程/数据）和 ELN（Electronic Lab Notebook，电子实验记录本）是实验室既有的「记账系统」。agent 落地绕不开和它们对接——这是壁垒也是入口。

C. MCP 与把它用到「仪器」上的设想

  • MCP（Model Context Protocol）：Anthropic 2024 年 11 月提出的开放协议，标准化「agent ↔ 工具/数据源」怎么对话，2025 年被 OpenAI/Google/Microsoft 广泛采纳，成了 agent 接工具的事实标准。modelcontextprotocol.io 🟢。但 MCP 设计的是「agent → 软件工具」这条边。
  • 把 MCP 用到仪器上为什么不够：仪器操作是有状态、安全攸关、独占占用、物理具身的——你不能像调用一个无状态 API 那样调用一台离心机或电镜。这正是 LAP / SCP 等新协议想补的「agent→仪器」这条边。
  • LAP 协议（arXiv 2606.03755）：🟢 已三源核实，真实存在。官方标题 「LAP: An Agent-to-Instrument Protocol for Autonomous Science」，2026-06-02 提交，cs.AI。提出四个物理世界原语：InstrumentCard（签名的能力/物理极限描述）、独占预约（exclusive reservation）、安全握手（safety-fence handshake，密码学绑定操作员确认 token，闸住危险/不可逆操作）、MeasurementResult（带单位 QUDT/UCUM、校准锚定、不确定度的结果 schema）。明确定位为补 MCP（agent↔工具）/ A2A（agent↔agent）未覆盖的 agent↔仪器层，并封装 SiLA2/OPC-UA。🟡 提示：2 周新的提案性预印本，无代码、无实现、无独立采用，别当成已落地标准。

D. 评测侧（Part B 的世界）

  • benchmark / leaderboard / arena：benchmark = 一组带标准答案的题，给模型打分；leaderboard = 把多个模型分数排成榜；arena（对战式评测）= 让两个模型同台答同一问题、人/裁判盲投谁更好，算 Elo（LMArena 是代表）。
  • eval-as-a-service：把「评测」做成一门生意——卖订阅看榜、卖第三方认证、卖定制评测咨询、或顺带卖评测用的数据。
  • 为什么科研 agent 评测比通用 LLM 更难：通用 LLM 评测的 ground-truth（标准答案）相对好搞（人标、考题）；但科研 agent 要验证「这个材料配方对不对 / 这个分子能不能合成」，ground-truth 往往要真做一次实验或跑一次昂贵仿真。这是 B 格商业化最大的死结：可信的答案太贵，便宜的答案不可信。

② Part A：科学仪器 MCP / agent-to-instrument 协议层

赛道定义

让 LLM/agent 真正「动手」操作实验仪器的中间件 / 协议 / 编排层——不是只生成 protocol 文本（那是 ChemCrow、Coscientist 的「大脑」部分），而是把指令落到真实设备：驱动移液机器人、HPLC、酶标仪、合成仪、电镜，处理排队/独占/安全/结果回传。可以把它类比成 AI4S 实验室的「操作系统 + 设备驱动 + MCP 总线」。这一层值钱，是因为它卡在「能跑实验的 agent」和「冷冰冰的厂商设备」之间，谁统一了它，谁就收过路费。

MCP 真的接到物理仪器了吗？（这格最关键的现状）

这是本格最值得讲清的一点：「真正用 MCP 驱动物理仪器」的案例屈指可数且极早期。市面上大多数叫「lab MCP」的 server，包的其实是数据库 / web API / 模拟器（UniProt、PDB、PubMed、Qiskit 仿真等），不是仪器，别混淆。按可信度排序：

Tier 1 — 真驱动物理仪器（最强证据）

  • TEM Agent / Foundry MCP（LBNL）🟢：arXiv 2511.08819「TEM Agent」（作者 Ercius 等，LBNL Molecular Foundry），用 MCP 把 LLM 接到 Molecular Foundry 的双球差校正透射电镜（经 Gatan DigitalMicrograph），用文本命令统一管理多个 TEM 子系统/数据/算力。国家实验室背书，最可信的真实仪器案例。
  • InstrMCP🟢：github.com/caidish/instrMCP，MCP 暴露 QCodes 仪器驱动 + JupyterLab 给 LLM，可读/操作量子器件实验室仪器；有 safe/unsafe 模式，MIT 许可，已测可配 Claude Desktop / Claude Code / Codex CLI。研究原型但工程成熟。
  • opentrons-mcp（社区）🟡：github.com/yerbymatey/opentrons-mcp，经 31950 HTTP API 真控 OT-2/Flex（上传协议、起停 run、归位等）。但个人 hobby 作品、非官方、无 license。值得注意：HighRes × Opentrons 在 SLAS 2026（2026-02 波士顿）用「Opentrons MCP server + OpentronsAI + HighRes Cellario」演示 agent-to-agent 生成 qPCR 工作流——这是最「工业级」的 MCP-in-loop 信号，但仍是会议 demo，非生产🟡。

Tier 2 — 公开承认 MCP 不够、提出扩展协议（早期 / 实现弱）

  • LAP — Lab/Agent-to-Instrument Protocol🟢：arXiv 2606.03755（2026-06，实验家 Shiyanjia Lab）。在 MCP/A2A 上加签名 InstrumentCard、预约/租约、加密 safety-fence 确认令牌、带单位与不确定度的 MeasurementResult；传输兼容 MCP/A2A，封装 SiLA2/OPC-UA。无代码、无实现。
  • SCP — Science Context Protocol（上海 AI Lab，SAIL）🟢：github.com/InternScience/scp，2025-12-30 开源（另见 arXiv 2512.24189）。MCP 的科研扩展，加中央 hub 编排、实验生命周期管理、JSON 实验上下文，连异构科研资源（软件/模型/数据集/工作流/实验室仪器）；配套 Intern-Discovery 平台称 1600+ 工具（生物 45.9% / 物理 21.1% / 化学 11.6%…）🔴 工具数为平台自述。真实仪器控制未公开演示🟡，但中国国家队已在卡这个位，对中国创业适配判断极重要。

对照 — 真自主、但不走 MCP 的成熟玩家

Artificial（SiLA+网关）、ECL（SLL）、Strateos（Autoprotocol JSON）、Synthace（Antha）、Ginkgo（Catalyst 脚本）都已解决「驱动级控制」的难点，但各用专有/非 MCP 路径。PyLabRobot（MIT 开源、硬件无关 SDK，真驱动 Hamilton/Tecan/Opentrons，但走代码生成、无 MCP 层）是最自然的「可 MCP 化」底座。

一句话定性：「科学仪器 MCP 协议层」的市场空位，正是——在 Artificial/ECL/Ginkgo/PyLabRobot 这些已经能控制设备的底座之上，补一层标准化的 NL/agent 接口。LAP / SCP 是这层的早期协议提案，TEM Agent / InstrMCP 是早期实现，但没有一个赢家、没有一个已落地标准。

早期度判档

nascent（萌芽），理由（信号）：

  • 标准林立未统一：SiLA2（gRPC 派）、OPC-UA LADS（工业 OPC 派）、Allotrope/AnIML（数据格式派）三摊并存；专为 agent 设计的 LAP（无实现）与 SCP（中国国家队，2025-12 才开源）同期冒头，谁都没赢。🟢
  • 「MCP 真接仪器」案例屈指可数：最硬的只有 LBNL TEM Agent 与 InstrMCP（学术/国家实验室），社区 opentrons-mcp 是 hobby 级，自称 production 的 QPillars 公开 demo 跑在数字孪生上。🟡
  • 巨头未通吃，但在押注：Microsoft（M12 投 Artificial）、NVIDIA（NVentures 投 Radical AI）、OpenAI（GPT-5 进 Ginkgo 自驱实验室）都从外围下注，但没有一家做出「仪器界的 MCP 总线」标准。🟡
  • 开源刚起：PyLabRobot（MIT，硬件无关 SDK，支持 Hamilton/Tecan/Opentrons）是最像「公共基础设施」的项目，但只几百 star 量级，远未到生态规模。🟢 github.com/PyLabRobot
  • 学术先例已证可行、但未平台化：A-Lab、Coscientist 证明了「agent 真能驱动仪器跑出结果」，但每套系统都从零重造 agent↔仪器的连接——这正是 LAP/SCP 论文点名的痛点，也是创业 whitespace 的来源。

国外成功案例拆解（商业模式怎么跑通）

案例 1：Emerald Cloud Lab（ECL）

做什么：「云实验室」——科学家在浏览器里用自研的 Symbolic Lab Language (SLL) 写实验，提交给 ECL 的机器人实验室跑，拿回结构化、可直接分析的数据。覆盖 200+ 仪器型号、已跑 60 万+ 真实实验。🟢

商业模式：订阅平台 access + 按实验/数据分析计费；用「一年全托管实验室的费用 < 买一台仪器的预付款」打动早期团队。🟡

能复刻的：「把仪器变成云 API + 订阅/按次计费」这条打法在中国可复刻（药企/CRO 愿为产能和复现性付费）。难复刻的：SLL 这种自研语言 + 全栈自营重资产，需要长期资本和仪器 know-how，不是软件团队能快速搭起来的。

案例 2：Ginkgo Bioworks（foundry 数据飞轮 + agent）

做什么：自营巨型 foundry（>$1B 自动化投入），用 Ginkgo Datapoints 卖「AI-ready 高质量生物数据」；2025 推 Cloud Lab + agent 栈：Catalyst Agent（对话式把意图转执行）、EstiMate（读自然语言 protocol、判可行性、给报价）、GPT-5 驱动自驱实验室。🟢

商业模式：foundry 服务 + 数据订阅 + Cloud Lab 按用量；护城河是数据飞轮。FY2025 已战略转向聚焦 Autonomous Labs。🟡

能复刻的：「自动化产能 → 标准化数据 → 卖数据/卖模型」的飞轮逻辑，对中国合成生物/材料国家队和大厂很有借鉴价值。难复刻的：需要十亿美元级重资产，早期团队碰不了——但可以做飞轮的「上层 agent / 编排层」那一段。

另一组可借鉴：Strateos × Eli Lilly 证明大药企愿为「远程云实验室」买单 🟢；Opentrons 证明「便宜开源硬件 + 自然语言生成 protocol（Opentrons AI）」能做成独角兽规模生态 🟡。

中国适配（四维）

维度判断说明
① 客户与付费方在不在中国 在，且预算实在 🟡 付费方齐全：创新药企/CRO（药明、康龙等产业链）、材料厂（电池/半导体）、高校与国家实验室（怀柔、合肥等大科学装置）、合成生物公司。痛点真实：招技术员贵、实验复现性差、产能瓶颈。预算量级：单个自动化实验室项目可达数百万至千万人民币级（设备+集成），药企/国家队付得起。镁伽 MegaRobo 已验证国内有这块市场。
② 国外成功模式哪条能复刻 部分能 🟡 能复刻：ECL 式「云实验室订阅 + 按实验计费」、Ginkgo 式「自动化→数据飞轮」的上层编排/agent 层、Strateos 式「给大药企做远程托管实验」。难复刻：从零自研一套全球标准（SLL/Autoprotocol 级）以及十亿美元级重资产 foundry——这些靠资本和时间，早期团队不碰。
③ 政策与数据壁垒 总体助力 🟡 助力：信创/国产替代直接利好「国产仪器编排中间件」（替代进口 SiLA/OPC 闭源栈）；国家实验室采购倾向国产；自主可控诉求强。更重要的信号：上海 AI Lab 已开源 SCP（Science Context Protocol），国家队亲自下场做「科研 MCP」，说明政策与生态资源在向这格倾斜——创业公司可借势（建生态/接 SCP）而非对抗。阻力：高端仪器（Hamilton/Tecan）进口受限——但这反过来是国产编排层的机会；生物样本/基因数据出境受《人类遗传资源管理条例》约束，做生物方向需注意。
④ 团队与技术门槛 高，但中国队拿得下 🟡 需要的不是顶级 research 血统，而是仪器/标准 know-how + 系统集成工程量 + 实验室运营经验——这恰是中国制造/自动化团队的强项。门槛在「懂设备协议 + 懂实验流程 + 能把 agent 接到物理世界并保证安全」，工程驱动，可控。比纯算法/模型创业更看苦活，但也更有壁垒、更难被大模型抹平。

这格最该切的 whitespace

切口：做「自驱实验室的编排/agent 落地层」，绑定国产仪器 + 行业垂直闭环交付，并接住 SCP/MCP 生态，而不是去死磕一个全新通用协议标准。

  • why-now：① MCP/agent 范式 2025 成熟，「让 agent 动手」从科幻变工程问题；② LAP/SCP/TEM Agent 等说明业界刚意识到 agent↔仪器这条边缺标准且尚无赢家——早入场能定义事实标准或吃生态红利；③ 国产仪器 + 信创替代给了「不依赖进口闭源栈」的政策窗口；④ 上海 AI Lab 的 SCP 给了中国创业者一个可对接的本土协议底座；⑤ 国家实验室/大科学装置正大规模自动化升级，是真实采购订单。
  • 具体打法：选一个垂直（如电池材料 / 抗体筛选 / 合成生物），做「自然语言/agent → 编排既有国产设备 → 安全执行 → 结构化数据回流」的整体交付，按项目+订阅+数据收费。技术上站在 PyLabRobot（开源）肩上，吸收 LAP 的「安全握手/独占预约/结果 schema」思想、接 SCP 生态，落地到国产设备。
  • 风险（诚实）：① 重——要碰物理硬件、现场集成、安全责任，交付周期长、毛利受集成拖累；② 标准未定，押错协议有返工成本；③ 巨头（镁伽、深势玻尔生态、大药企自建、上海 AI Lab 顺手下沉）可能挤压；④ 客户决策慢（国家队采购周期长）。对冲：先做「软件编排层 + agent」轻资产切入，硬件用合作伙伴的，验证 PMF 后再决定要不要重。

③ Part B：科学 agent 评测即服务（eval-as-a-service）

赛道定义

给 AI4S 模型 / 科研 agent 做 benchmark、leaderboard、arena、第三方评测与认证——卖「可信度量」。买方逻辑：当药企/材料厂要采购或信任一个科研 agent 时，需要一个中立可信的「它到底行不行」的度量。这格的核心矛盾在背景速补已点明：科研 agent 的 ground-truth 太贵（要真做实验或跑昂贵仿真），便宜的答案（人标选择题、文献 QA）又不够说明「能不能真发现」。

早期度判档

科研专用评测：nascent（几乎无人商业化）；通用 eval：emerging→crowded（作对比）。信号：

  • 科研专用评测全是学术开源、零商业化：上表 11 个 benchmark 无一靠卖评测赚钱，全是论文产出物；连最活跃的私营玩家 FutureHouse（占 LAB-Bench/BixBench 两个）也是慈善资助、不靠评测变现。🟢
  • 通用 eval 已挤：数据+eval 巨头 Scale($29B)/Surge/Mercor($10B) 估值高企；独立横评（Artificial Analysis）、非营利评测（METR/Epoch）、开发者 eval 工具（Galileo/Patronus/Braintrust/Arize）各占位；连最中立的 HF Open LLM Leaderboard 都因「基准失效」在 2025-03 退役——说明「卖一个静态榜单」本身不可持续。🟢
  • 真正能持续的评测形态有两种：① 绑死「数据/标注劳动力」（Scale/Surge/Mercor，估值来自数据生意而非「评测」本身）；② 非营利/中立公共品（METR/HF）。纯粹靠「卖科研评测度量」当独立生意，目前全球都没跑通。

国外成功案例拆解

案例 1：Scale AI —「评测是数据生意的尾巴」

Scale 真正赚钱的是数据标注，评测（SEAL 榜、红队）是为了卖更多数据、增强客户黏性的附加层。Meta 2025-06 以 ~$14.3B 拿约 49%、估值 ~$29B。🟢

启示：评测本身难单独卖，但「评测 + 数据/标注劳动力」能撑起巨大估值。中国能复刻「数据生意」（已有数据标注产业），但「独立第三方评测」这层在中国客户付费意愿弱。

案例 2：LMArena —「中立榜单 → 公司化」的样板与陷阱

LMSYS 的 Chatbot Arena 用「盲投对战 + Elo」做出全球最有公信力的 LLM 榜，2025 公司化（Arena Intelligence），种子 $100M @ $600M（a16z+UC）。🟢 但商业化路径仍待定，且公司化后中立性遭论文质疑。

启示：arena 模式（社区盲投）是少数能持续运营的评测形态，对「科研 agent arena」有借鉴价值。但「公信力」与「商业化」天然冲突——一旦收钱，中立性受损。

中国适配（四维）

维度判断说明
① 客户与付费方在不在中国 付费方弱/不清晰 🔴 谁为「科研 agent 评测」付费？药企/材料厂买的是结果（药/材料），不是「评测报告」；模型公司（百度/智谱/月之暗面）会自建评测、不愿外包；国家实验室/高校做评测是科研产出（发论文/拿影响力），不是采购。结论：缺真实的、预算明确的 B 端付费方。中国三家（OpenCompass/FlagEval/SuperCLUE）全是机构非营利，恰好印证。
② 国外成功模式哪条能复刻 只有「数据生意」那条能 🟡 Scale 路径中能复刻的是数据标注/数据生成（中国有成熟数据标注产业），但那本质是 Part A 式「数据飞轮」或数据外包，不是「评测即服务」。LMArena 的 arena 模式可借鉴形态，但变现难。METR/Epoch 的非营利路线在中国对应 OpenCompass/FlagEval——靠机构养，不是创业生意。
③ 政策与数据壁垒 中性偏阻力 🔴 「独立第三方评测/认证」在中国更可能由官方/国家队主导（信通院、上海 AI Lab、智源），留给创业公司的「中立认证」空间小——公信力背书往往需要官方身份。科研数据合规反而增加做评测的数据获取成本。
④ 团队与技术门槛 技术不高，但「公信力」门槛极高 🟡 做一个 benchmark 的工程量不大（学术界一篇论文就能产出），真正的壁垒是公信力/中立性背书——这恰恰是创业公司最难自证、且最容易被「既当裁判又当选手」质疑的（LMArena/Scale 都中过招）。对科研评测还要叠加领域 ground-truth 的获取（昂贵）。

这格最该切的 whitespace（诚实：偏小）

切口：不要把「科研评测」当独立生意卖，而是把它做成 Part A（自驱实验室）或某个垂直 AI4S 产品的嵌入能力——用「真实实验/仿真 ground-truth」做出别人做不出的评测，反哺自己的模型/服务。

  • why-now：① 全球没有一个 benchmark 用真实湿实验做 ground-truth——谁有自驱实验室，谁就能做出「真·可信」的科研 agent 评测，这是 A×B 的天然协同；② 材料/原子势方向已有持续运营的 arena 形态（Matbench Discovery / MLIP Arena / LAMBench），中国 DP 生态已在 LAMBench 占位，可顺势做「材料 agent arena」。
  • 可行的小生意：给特定行业（如药物/材料）做「闭源 holdout + 真实实验回测」的认证/排名，但更可能是 To 政府/联盟的项目，而非可规模化的 SaaS。
  • 风险（诚实）：① 付费方不明确，独立做大概率饿死；② 公信力靠官方背书，创业公司天然弱势；③ 容易被模型公司/巨头顺手做掉（自带评测）。结论：作为创业主轴不推荐；作为 A 格的护城河组件或学术影响力工具，价值高。

④ 合并结论：两格对比 + 给中国创业者的取舍

明确结论

  1. 哪一格更适合中国早期创业 → Part A（仪器 MCP / 自驱实验室编排层）。三条硬理由：(1) 付费方真在中国且预算实在（药企/CRO/材料厂/国家实验室）；(2) 政策是助力（信创国产替代 + 国家实验室自动化升级采购，且上海 AI Lab 已开源 SCP 给出本土协议底座）；(3) 国外成功商业模式清晰可借鉴（ECL 云实验室订阅+按次计费、Ginkgo 数据飞轮 + Catalyst/EstiMate agent、Strateos 大药企远程托管）。门槛是仪器 know-how + 工程量——这是中国自动化/制造团队的主场，不是劣势。
  2. Part B（科研评测）不建议作为独立创业主轴：全球都没跑通「卖科研评测度量」的独立商业模式，中国付费方更不明确、公信力又靠官方背书。正确姿势：把评测做成 Part A 的护城河组件（用真实实验 ground-truth 做出别人做不出的评测），或作为模型/垂直产品的内部能力。
  3. 两格的天然协同：B 格最大空白是「没有 benchmark 用真实湿实验做 ground-truth」，而 A 格恰好能产生真实实验数据——做 A 格的团队顺手就能做出 B 格做不出的可信评测。这是「先 A 后 B」的正确顺序，反过来不成立。
