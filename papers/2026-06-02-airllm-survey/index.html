一句话 — AirLLM 让你在极小显存（4GB 跑 70B、8GB 跑 Llama-3.1 405B）上做大模型推理，靠的是 逐层加载（layer-by-layer）+ disk offloading：一次只把一层权重搬进 GPU 算完即卸，峰值显存 ≈ 一层（70B 约 1.6GB），不靠量化/蒸馏、保持原始精度。代价是非常慢——每个 forward 都要把所有层从磁盘读一遍，等于每出一个 token 就把整模型读一遍。定位是「能不能在烂显卡上跑起来」的实验工具，不适合交互/生产。真要日常用，llama.cpp（量化）/ ktransformers（MoE offload）/ PowerInfer（稀疏） 几乎总是更实际。

①它是干嘛的

定位在消费级小显存上对超大模型做推理（70B@4GB、Llama-3.1 405B@8GB），不量化、不蒸馏、不剪枝，保持原模型精度
用途纯推理（非训练）、单机；给「显卡弱但想跑大模型」的研究/猎奇/离线场景
支持Llama 2/3/3.1、Qwen、ChatGLM、Baichuan、Mistral、InternLM、Mixtral 等；Linux / macOS(Apple Silicon) / CPU
额外可选 4/8-bit 分块压缩（仅压权重、走 bitsandbytes），官方称最高 ~3× 提速
APIAutoModel.from_pretrained(...) + .generate(...)，接口极简
性质仓库 96% 是 Jupyter Notebook —— 偏 demo / 研究演示，非生产级硬核库

②核心技术：逐层加载 + disk offloading

Transformer 是一层层堆起来的，层 N 只依赖层 N-1 的输出，并不需要所有层同时在显存里。AirLLM 把这点用到极致：

  • 切层：初始化时把 checkpoint 按 transformer 层切成一片片存盘。
  • 逐层前向：加载层 N 到 GPU → 算出激活 → 卸载层 N → 预取层 N+1。峰值显存从 140GB+ 降到 <4GB（砍 95%+）。
  • 不量化也能塞下：因为「塞不下」的根因是一次性放整模型，逐层之后每次只放一层，于是全精度也行——这是它对比 llama.cpp（要量化）的卖点：零精度损失。

③慢在哪 / 适用场景

硬伤：慢，而且是结构性的慢 自回归生成每出一个 token 就是一次完整 forward，而一次 forward 要把所有层从磁盘搬一遍。所以 AirLLM 约等于「每个 token 都把整个 70B 模型从硬盘读一遍」——README 也明说「bottleneck is mainly at the disk loading」。不压缩时常见 每 token 数十秒~分钟级；磁盘最好是 NVMe。

因此它适合：① 在没有大显卡的机器上「能不能跑起来」的验证 / 学习；② 对单次、离线、不在乎延迟的批处理（如跑一批离线评测、生成少量样本）；③ 想要全精度、零量化损失地试某个超大模型。
不适合：交互聊天、在线 serving、任何在乎吞吐/延迟的场景；也不能用于训练。

④同类竞品对照（按「怎么把大模型塞进小硬件」分四类）

A. 同思路：权重 offload + 保持全精度（AirLLM 的直接竞品）

项目做法vs AirLLM
HF Accelerate（device_map="auto" + disk/CPU offload）自动把放不下的层 offload 到 CPU/磁盘最主流的「官方等价物」，更通用稳定；同样慢，不支持跨机
DeepSpeed ZeRO-Inference整模型权重 offload 到 CPU/NVMe，支持 4-bit 量化 + KV cache offload更工程化、面向 NVMe；比 AirLLM 成熟
FlexGen（Stanford, 2023）单 GPU offload，但为大 batch 吞吐优化批吞吐远高；但延迟差、偏 OPT 老模型、维护淡

B. 最实际的替代：量化塞进显存（大多数人该走这条）

项目做法 / 说明
llama.cpp / GGUF（+ Ollama / LM Studio / Jan）Q4 等量化 + -ngl 部分层上 GPU + mmap，跑大模型于小硬件的事实标准，快得多；代价是量化有微小质量损失
bitsandbytes 4/8-bit · ExLlamaV2/V3 · AWQ/GPTQ · MLC-LLM各种量化把模型压进 VRAM；ExLlama 在 GPU 上又快又省，MLC 面向 edge/移动/web

C. 激活稀疏 GPU-CPU 混合

PowerInfer / PowerInfer-2：把「热神经元」放 GPU、「冷神经元」放 CPU，按激活稀疏只算该算的，消费级 GPU 上明显更快。

D. MoE 专用 offload / 分布式

  • ktransformers：超大 MoE（如 DeepSeek）在有限显存 + 大内存 CPU 上跑，体验通常比 AirLLM 好一个量级——跑 MoE 首选。
  • mixtral-offloading：Mixtral 类 MoE 的专家级 offload。
  • Petals：BitTorrent 式把 100B+ 拆到多台志愿者机器上协同推理——没有一块够大的卡、但有多台机器时用。
一句话区分 AirLLM / Accelerate / ZeRO-Inference / FlexGen = 「不量化，靠 offload 硬塞」（保精度但慢）；llama.cpp / ExLlama / bitsandbytes / MLC = 「量化压进显存」（快但有微损）；PowerInfer = 「稀疏只算一部分」；Petals = 「多机分摊」。

⑤怎么选（结论）

  • 只想「在烂显卡上把某大模型跑起来看看」且不要任何量化损失 → AirLLM 或 Accelerate disk-offload，接受龟速。
  • 想实际用得动（聊天/写代码） → 直接 llama.cpp / GGUF（Ollama / LM Studio），量化后又快又省，90% 场景的正确答案。
  • 跑超大 MoE（DeepSeek/Mixtral）且有大内存 CPU → ktransformers。
  • 要高吞吐批处理而非低延迟 → FlexGen / ZeRO-Inference。
  • 没有够大的卡、但有多台机器 → Petals。
放进大图 「在小硬件上跑大模型」本质两条路：靠「省」（offload / 量化 / 稀疏，AirLLM 属此）与 靠「堆」（大统一内存，如 NVIDIA RTX Spark / DGX Spark 的 128GB）。AirLLM 是「省」到极致的演示——聪明、零精度损失，但用磁盘 IO 换显存，慢是必然代价。
